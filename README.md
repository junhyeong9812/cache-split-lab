# cache-split-lab

> 캐시를 가진 티어를 복제하면 무슨 일이 일어나는가.

📖 **처음이라면 [docs/GLOSSARY.md](docs/GLOSSARY.md) 부터.**
LRU · Zipf · arm · 워킹셋 · 히트율 등 이 문서에 나오는 용어를 처음 보는 사람 기준으로 설명한다.

## 질문

서버가 버거워졌을 때 선택지는 둘이다 — **크게 키우거나(scale-up), 한 대 더 띄우거나(scale-out).**
총 자원이 같다면 보통 둘은 비긴다. 그런데 **앱이 인메모리 캐시를 들고 있으면 비기지 않는다.**

노드를 2대로 쪼개면 노드당 캐시가 절반이 된다. 그리고 라운드로빈 로드밸런서 아래에서는
**두 노드가 각자 전체 키 공간을 보면서** 각자의 LRU로 "가장 뜨거운 절반"을 남긴다.
둘이 보는 트래픽 분포가 같으니 **둘 다 똑같은 키를 캐싱한다.**
시스템 전체 캐시 용량은 그대로인데 실제로 커버하는 *서로 다른* 키는 절반이다.

이게 낭비인지 아닌지는 **접근 분포가 정한다.**

- 롱테일 카탈로그 탐색(평평한 분포) → 중복은 **낭비**다. 메모리 절반을 복제본에 태운다.
- 플래시 세일(극단적 쏠림) → 중복은 **기능**이다. 뜨거운 키는 어차피 모든 노드에 들어가고,
  오히려 모든 노드가 그 키를 가져야 아무 노드나 응답할 수 있다.

그래서 이 프로젝트의 명제는 "스케일아웃하면 캐시가 깨진다"가 아니라 이것이다:

> **중복 캐시는 낭비인가 복제인가 — 그건 접근 분포가 정한다.
> 그리고 그 경계가 어디인지는 돌려봐야 안다.**

## 실험군

총 캐시 용량을 **셋 다 동일하게** 고정하고 배치만 바꾼다. 이게 공정 비교의 전부다.

| Arm | 배치 | 역할 |
|-----|------|------|
| **A** `a-single` | 1 노드 × 캐시 N | **자(ruler).** 캐시를 쪼개지 않았을 때의 천장 |
| **B** `b-roundrobin` | 2 노드 × 캐시 N/2, nginx 라운드로빈 | 함정 재현 |
| **C** `c-keyhash` | 2 노드 × 캐시 N/2, nginx 키 해시 | 함정 해소 — 그리고 그 대가 |

**A는 네 번째 경쟁자가 아니라 자다.** 히트율 71%가 좋은 건지 나쁜 건지는
"쪼개지 않았으면 몇 %였나"를 알아야 말할 수 있다. A가 없으면 함정의 크기를 측정할 수 없다.

**nginx는 A에도 물린다.** A만 LB를 건너뛰면 네트워크 홉 하나를 공짜로 아끼게 되어
"캐시 효과"와 "LB 비용"이 섞인다. 홉을 상수로 고정해야 배치만 변수로 남는다.

**앱 소스는 하나다.** `app/` 하나를 셋이 공유하고 arm 별로 다른 건 설정뿐이다.
arm 마다 코드를 복사하면 셋이 조금씩 달라지는 순간 실험 전체가 무효가 된다 —
"arm 간 차이는 배치뿐"이 이 실험의 최상위 불변식이고, 구조가 그걸 강제해야 한다.

## 축

| 축 | 범위 | 왜 |
|----|------|-----|
| **워킹셋 ÷ 캐시** | 0.5 → 8배 | 함정이 나타나는 창이 좁다. 워킹셋 ≪ 캐시면 둘 다 100%, ≫ 캐시면 둘 다 0%. **≈ 부근에서만 터진다** |
| **Zipf skew** | 평평 → 극단 | 여기서 **B와 C의 우열이 뒤집힌다.** skew ↑ → 캐시 함정은 약해지고(핫키는 어디든 들어감) 핫키 쏠림은 심해짐(C의 한 노드가 다 받음) |

**교차점을 찾는 게 이 프로젝트의 산출물이다.** 그 점은 계산으로 유도되지 않는다 —
LRU 축출 동역학과 실제 부하 쏠림이 얽혀 있어서 돌려봐야 안다.

## 측정하는 것

- **히트율** (arm 별 · A 대비 손실) ← 주 지표. 카운터라서 CPU 경합에 안 흔들린다
- **노드별 요청 분포** ← C의 핫키 쏠림. 이게 없으면 C의 대가를 못 본다
- p50 / p95 / p99 지연, 처리량, 에러율

## 물리 구성

```
APC  192.168.55.9    부하 생성 (k6)
BPC  192.168.55.164  SUT (nginx + 앱 인스턴스 + origin)
```

부하 생성기는 **반드시 SUT와 물리적으로 분리**한다. 같은 머신에 두면 부하가 커질수록
부하 생성기가 CPU를 뺏어서, 측정된 "포화 지점"이 실은 부하 생성기의 포화 지점이 된다.

자세한 구성은 `docs/ARCHITECTURE.md`.

## 안 하는 것

- **개인화 캐시** — 업계는 개인화를 캐싱하지 않고 캐시 경로 밖으로 뺀다(fragment 캐싱 · 클라이언트
  사이드 개인화 · 세그먼트 단위 키). 우리 캐시는 **객체 캐시**(`product:42`)라 개인화가 들어올 자리가 없다.
- **개인화 비율 축** — 히트율 = `(1-x) × 기본히트율`. **직선이다.** 유도되는 걸 측정하지 않는다.
- **게이트웨이 캐싱 arm** — nginx `proxy_cache`를 1대로 두면 A의 천장을 회복하고(당연 — 안 쪼갰으니),
  2대로 두면 함정이 그대로 재현된다(당연 — B와 같은 메커니즘이 한 층 위에서). 둘 다 연역되므로
  **실험이 아니라 시연**이다. 런 매트릭스가 아니라 문서에 들어간다.

기준: **유도할 수 있는 건 유도하고, 유도 못 하는 것만 측정한다.**

## 구조

```
app/                단일 앱 소스 — 3 arm 이 같은 이미지를 쓴다
  src/…/web/TestApiController.kt    /key/{id} · /admin/stats · /admin/cache/keys
origin/             백킹 스토어. CSV 인메모리 + 고정 지연
nginx/
  nginx.conf        공통 — 세 arm 이 그대로 공유
  upstream/
    arm-a.conf      노드 1개
    arm-b.conf      노드 2개 라운드로빈
    arm-c.conf      노드 2개 hash $uri     ← arm 간 유일한 차이
arms/
  compose.yml       하나. 세 arm 이 공유한다
  a-single/arm.env      ARM_UPSTREAM_CONF=arm-a.conf, 노드 1개
  b-roundrobin/arm.env  ARM_UPSTREAM_CONF=arm-b.conf, 노드 2개
  c-keyhash/arm.env     ARM_UPSTREAM_CONF=arm-c.conf, 노드 2개
data/               생성된 데이터셋 (CSV — 추적 안 함)
k6/                 부하 시나리오 (APC 에서 실행)
scripts/            스윕 러너 · 배포
results/            실험 산출물 (추적 안 함 — 재생성 가능)
docs/               ARCHITECTURE.md · GLOSSARY.md
```

**arm 폴더에 코드가 없는 게 핵심이다.** compose 도 앱도 하나뿐이고, arm 이 바뀌면
`arm.env` 가 바뀔 뿐이다 — 파일이 하나면 구조적으로 갈라질 수가 없다.
"셋은 설정만 다르다"가 이 프로젝트의 명제이자 폴더 구조다.

### 실행

```bash
./gradlew build                          # jar 를 먼저 만든다 (셋이 같은 jar 를 쓴다)
scripts/run-arm.sh a-single              # ← 미구현. 총량에서 노드당 값을 계산해 주입한다
```

compose 를 직접 부르면 `:?` 가드에 걸려 즉시 실패한다. 의도적이다 —
빠뜨린 캐시 크기가 기본값으로 조용히 채워지면 실험이 통째로 거짓말이 된다.
