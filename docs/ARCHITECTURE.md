## 1. 물리 구성

```
┌─────────────────────────────┐         ┌──────────────────────────────────────────────┐
│  APC   192.168.55.9         │         │  BPC   192.168.55.164                        │
│  (부하 생성)                 │         │  (SUT — System Under Test)                   │
│                             │         │                                              │
│   ┌───────────────┐         │  LAN    │   ┌────────────┐                             │
│   │      k6       │─────────┼────────►│   │   nginx    │  :80                        │
│   │  Zipf 키 생성  │         │  HTTP   │   │  (arm 별   │                             │
│   │  일정 도착률    │         │         │   │   설정 교체)│                             │
│   └───────────────┘         │         │   └─────┬──────┘                             │
│                             │         │         │                                    │
└─────────────────────────────┘         │         ▼                                    │
                                        │   ┌──────────────┐                           │
   부하 생성기는 반드시 분리한다.          │   │  app 노드     │  캐시(LRU) 보유           │
   같은 머신이면 부하가 커질수록           │   │  1~2 개       │                           │
   생성기가 CPU 를 뺏어서, 측정된          │   └─────┬────────┘                           │
   "포화 지점"이 실은 생성기의             │         │ miss 시에만                        │
   포화 지점이 된다.                      │         ▼                                    │
                                        │   ┌──────────────┐                           │
                                        │   │   origin     │  백킹 스토어                │
                                        │   │  (미스 페널티) │  고정 지연 + 값 반환        │
                                        │   └──────────────┘                           │
                                        └──────────────────────────────────────────────┘
```

**origin 은 실제 DB 가 아니다.** CSV 를 통째로 메모리에 올려두고 **고정 지연 후 반환**한다.
전부 메모리에 상주하므로 **미스 비용이 모든 키에 대해 균일**하고, 우리가 정한 값 그대로다.

실 DB(H2 등)를 붙이면 단순히 노이즈가 느는 게 아니라 **효과를 체계적으로 상쇄한다**:
DB 는 자기 버퍼 캐시를 갖는데, **그 캐시 상태가 arm 마다 다르게 데워진다.**
B 는 미스를 많이 내니 DB 캐시를 더 넓게 데우고 → **B 의 미스가 A 의 미스보다 싸진다**
→ 우리가 재려는 "B 의 손해"를 DB 가 몰래 깎아준다. 노이즈면 평균으로 없어지지만
이건 **편향**이라 안 없어지고, 하필 방향이 우리 결론을 약하게 만드는 쪽이다.

**미스 페널티는 파라미터여야지 측정 대상이 아니다.** 그래야 arm 간 차이가
순수하게 미스 **횟수** 차이로만 남는다.

## 2. Arm 별 배치

**변하는 것은 배치뿐이다.** 앱 이미지·origin·k6 시나리오·요청 시퀀스(동일 seed)는 전부 상수.

```
Arm A — a-single ················ 자(ruler). 캐시를 쪼개지 않은 천장.

     k6 ──► nginx ──► [ app-1 : 캐시 N ] ──► origin
                       (upstream 1개)

     nginx 가 분산할 게 없어도 경로에 둔다.
     A 만 홉을 건너뛰면 "캐시 효과"와 "LB 비용"이 섞여 분리가 안 된다.


Arm B — b-roundrobin ············ 함정. 두 노드가 같은 키를 중복 캐싱.

                    ┌──► [ app-1 : 캐시 N/2 ] ──┐
     k6 ──► nginx ──┤      (round-robin)        ├──► origin
                    └──► [ app-2 : 캐시 N/2 ] ──┘

     두 노드가 각자 전체 키 공간을 본다 → 각자 LRU 로 "가장 뜨거운 절반"을 남김
     → 둘이 같은 트래픽 분포를 보므로 같은 키를 캐싱 → 유효 캐시 = N/2


Arm C — c-keyhash ··············· 해소. 키 공간을 분할 소유.

                    ┌──► [ app-1 : 캐시 N/2 ] ──┐   담당: hash(key)%2==0
     k6 ──► nginx ──┤   hash $arg_key           ├──► origin
                    └──► [ app-2 : 캐시 N/2 ] ──┘   담당: hash(key)%2==1

     각 노드가 자기 몫 키만 본다 → 중복 없음 → 유효 캐시 = N
     대가: 핫키가 한 노드에 쏠린다. skew 가 높을수록 심해진다.
```

### 상수 / 변수

| | |
|---|---|
| **상수** | 앱 이미지 · origin 지연 · k6 시나리오 · 요청 시퀀스(동일 seed) · nginx 홉 존재 · **총 캐시 용량** |
| **변수** | replica 수 · 노드당 캐시 크기 · nginx 라우팅 방식 |
| **축** | 워킹셋 ÷ 캐시 · Zipf skew |

**총 캐시 용량 고정(A: 1×N, B/C: 2×N/2)이 깨지면 비교 자체가 무효다.**

## 3. 결정 — 캐시는 바이트가 아니라 **엔트리 개수**로 잰다

"캐시 2GB"를 바이트로 재려 하면 실험이 오염된다:

- 런타임마다 "N개 엔트리 = 몇 바이트"가 다르다 (객체 헤더 · 정렬 · 내부 자료구조 오버헤드)
- JVM 계열이면 힙 + GC 여유 + 메타스페이스가 컨테이너 리밋을 넘겨 **OOMKill**
- 결국 arm 간 "같은 캐시 크기"를 보장할 수 없다 → **최상위 불변식이 깨진다**

**"캐시는 5,000 엔트리를 보관한다"는 정확하고 언어 무관하다.** 워킹셋도 키 개수로 세므로
축(워킹셋 ÷ 캐시)이 **무차원 비율**이 되어 해석도 깔끔해진다.

그 결과 **도커 메모리 리밋은 실험의 메커니즘이 아니라 안전망**이 된다.
cgroup 리밋은 "터지면 알려주는 장치"로만 두고, 캐시 크기는 앱 파라미터로 강제한다.

> 이 결정의 부작용: 메모리 압박 자체(GC·스왑·OOM)는 이 실험의 대상이 아니게 된다.
> 그건 다른 실험이다. 여기서는 **히트율의 논리**만 잰다.

## 4. Arm 실행은 순차다

세 arm 을 동시에 띄우면 BPC 한 대의 CPU·메모리·NIC 을 셋이 나눠 갖게 되어
**서로가 서로의 노이즈**가 된다. arm 하나씩 올리고 → 워밍업 → 측정 → 내리고 → 다음.

nginx 설정은 arm 별로 교체한다(포트 분리 아님 — 포트를 나누면 동시 기동을 유혹한다).

## 5. 측정

| 지표 | 어디서 | 왜 |
|------|--------|-----|
| **히트율** (arm 별, A 대비 손실) | 앱 노드 카운터 | **주 지표.** 카운터라 CPU 경합에 안 흔들린다 |
| **노드별 요청 수** | nginx 또는 앱 | C 의 핫키 쏠림. 없으면 C 의 대가를 못 본다 |
| **노드별 캐시 키 집합** | 앱 덤프 엔드포인트 | **B 의 중복을 직접 증명.** 교집합 크기가 곧 낭비량 |
| p50/p95/p99 · 처리량 · 에러율 | k6 | 부수 지표 |

**`노드별 캐시 키 집합`이 이 실험의 스모킹 건이다.** 히트율 차이는 "결과"고,
두 노드 캐시의 **교집합**은 "원인"이다. B 에서 교집합 ≈ 100%, C 에서 ≈ 0% 가 나와야
우리가 측정한 게 진짜 그 메커니즘이라는 증거가 된다.

## 6. 검증 (sanity)

숫자가 나오기 전에 벤치 자체가 맞는지부터 증명한다. **이게 없으면 모든 숫자가 의심스럽다.**

- [ ] 캐시 = 워킹셋 전체 → A·B·C 모두 히트율 ≈ 100%
- [ ] 캐시 ≈ 0 → A·B·C 모두 히트율 ≈ 0%
- [ ] B 의 두 노드 캐시 교집합 ≈ 100% (중복 실재 증명)
- [ ] C 의 두 노드 캐시 교집합 ≈ 0% (분할 실재 증명)
- [ ] C 의 키→노드 매핑이 결정적 (같은 키 반복 요청 → 항상 같은 노드)
- [ ] 같은 seed → 같은 요청 시퀀스 (arm 간 비교 가능성의 전제)
- [ ] Zipf 생성기가 실제로 Zipf (생성된 시퀀스의 분포를 직접 검정)

**이 실험의 지배적 실패모드는 "조용히 틀린 숫자"다.** 히트율 71%는 맞아도 71%,
틀려도 71%처럼 보인다. 카운터 레이스 하나, LRU 축출 버그 하나, Zipf 아닌 Zipf 생성기 하나면
그럴듯하고 확신에 찬 거짓 결론이 나오고 — **틀린 교훈을 배우는 건 아무것도 안 배우는 것보다 나쁘다.**

### #1 Kotlin 채택 근거
앱이 하는 일이 HTTP 핸들러 + LRU + 카운터뿐이라 **언어에서 배울 게 없다.**
그러면 익숙한 쪽이 낫다 — 언어와 씨름하는 시간은 이 프로젝트에 순손실이다.

> JVM 의 JIT 워밍업이 교란 변수가 될까 검토했으나 **기각**: 이 실험은 캐시 워밍업이
> 어차피 10⁵~10⁶ 요청 규모인데 C2 컴파일 임계치는 10⁴ 규모라 **긴 쪽이 짧은 쪽을 덮는다.**
> 게다가 주 지표인 히트율은 JIT 의 영향을 아예 안 받는다(JIT 는 속도를 바꾸지
> 캐시에 어떤 키가 들었는지를 바꾸지 않는다).

### #2 직접 짠 LRU 채택 근거
Caffeine 을 안 쓰는 이유는 **무거워서가 아니다** — Caffeine 은 가볍고 빠르다. 이유는 셋:

1. **Caffeine 은 LRU 가 아니다.** W-TinyLFU 를 쓴다 — **Zipf 쏠림에서 LRU 보다 잘하도록
   설계된** 정책인데, 하필 우리 축이 Zipf 쏠림이다. 재는 대상이 바뀐다.
   (단 **중복 현상 자체는 정책 무관**하다 — 라운드로빈이면 두 노드가 같은 분포를 보므로
   어떤 정책이든 같은 핫셋으로 수렴한다. 바뀌는 건 곡선의 모양과 해석 가능성이다.)
2. **크기가 근사치다.** `maximumSize(n)` 은 비동기 축출·세그먼트 구조 때문에 정확하지 않다.
   최상위 불변식이 "총 캐시 용량 고정"인데 근사면 곤란하다.
3. **분석이 안 된다.** ← 결정타. 아래 참조.

### #3 origin 채택 근거
§1 참조 — DB 의 내부 캐시가 arm 마다 다르게 데워져 **효과를 체계적으로 상쇄**하기 때문.

## 8. 남은 공백 — 기준소스

**"실물에서 나온 숫자가 맞는지 무엇으로 검증하나?"** 시뮬레이터를 보류하면서 생긴 구멍이다.

**LRU 를 직접 짜기로 한 결정(#2)이 이 구멍을 메울 수 있다.**
LRU + Zipf 조합은 **이론 히트율을 해석적으로 근사할 수 있다**(Che's approximation —
IRM 가정 하의 LRU 히트율 모델, CDN/캐싱 문헌의 표준 도구). 그러면 매 지점마다
**"이론값 vs 실측값"** 대조가 가능해지고, 그게 곧 기준소스가 된다.
W-TinyLFU 에는 이런 닫힌 형태가 없다 — Caffeine 을 골랐다면 이 길이 막혔다.

> ⚠️ **미검증**: Che's approximation 의 정확한 수식은 아직 출처로 확인하지 않았다.
> 기준소스로 채택하기 전에 원문을 찾아 확인할 것. 확인 전까지 이 절은 **계획이지 근거가 아니다.**

그때까지는 §6 sanity 체크가 부분적으로 그 역할을 대신한다.

## 9. 후속 실험 (지금 섞지 않는다)

### 콜드 스타트 — 스케일아웃의 진짜 비용
스케일아웃으로 노드를 새로 띄우면 그 인스턴스는 **캐시도 차갑고 JIT 도 차갑다.**
부하를 못 견뎌 노드를 추가했는데 **새 노드가 한동안 제일 느린 노드**가 되고,
LB 는 그것도 모르고 트래픽을 똑같이 나눠준다 → 스케일아웃 직후 오히려 p99 가 튄다.
실무에서 오토스케일링이 배신하는 대표적 이유.

지금 리그 위에 나중에 그대로 올릴 수 있다. **지금 섞으면** arm 간 지연 차이가
캐시 배치 때문인지 JIT 상태 때문인지 구분이 안 된다.

> 주의: 로컬에선 컨테이너가 1초면 떠서 실제 인스턴스 기동 지연(1~2분)이 재현되지 않는다.
> 이 실험을 할 거면 기동 지연을 인위적으로 넣어야 공정하다.

### 게이트웨이 캐싱의 재귀
nginx 를 2대로 늘리고 각자 `proxy_cache` 를 로컬로 두면 **같은 함정이 한 층 위에서
그대로 재현된다.** 함정은 "게이트웨이냐 앱이냐"가 아니라 **"로컬 캐시를 가진 티어를
복제하는가"**의 문제이기 때문. 다만 이건 §1 의 메커니즘에서 연역되므로
**실험이 아니라 시연**이다 — 런 매트릭스가 아니라 `TECHNICAL.md` 에 들어간다.
