> **이 문서는 "지금 설계가 무엇인가"가 아니라 "왜 이렇게 됐고 무엇을 버렸는가"를 남긴다.**

> 좋지 않나?" 를 다시 제안하게 되는 걸 막는 게 아니라 — **그때 이유를 읽고
> 뒤집을지 말지 판단할 수 있게** 하는 것이다. 결정은 뒤집혀도 되지만

## 여정 요약

```
"스케일업 vs 스케일아웃을 컴퓨터 3대로 실험하고 싶다"
   │
   ├─ 워크로드 3종(CPU-bound / 캐시 / DB-bound) 중
   │  캐시 워크로드에서만 스케일업이 압승 → 여기가 제일 안 알려진 함정
   │
   └─→ "인메모리 캐시를 가진 티어를 복제하면 무슨 일이 일어나는가" 하나로 축소
          │
          ├─ 명제 교정: "스케일아웃하면 캐시가 깨진다" (틀림)
          │            → "중복 캐시는 낭비인가 복제인가 — 분포가 정한다" (맞음)
          │
          └─ 축: 키 공간÷캐시 × Zipf skew.  arm: A(자) / B(RR) / C(키해시)
```

## 1. 스코프 — 캐시 함정 하나로 축소

**결정.** 스케일업/아웃 일반이 아니라 **"로컬 캐시를 가진 티어를 복제할 때"** 하나만 다룬다.

**근거.** 총 자원이 같으면 스케일업과 스케일아웃은 대체로 비긴다. 안 비기는 데가
캐시다 — 그리고 그게 교과서에 잘 안 나오면서 현업에서 자주 물리는 지점이다.

**버린 것.**
- CPU-bound 워크로드 (해시·이미지 리사이즈) — 스케일업 ≈ 스케일아웃. 대조군으로는
  의미 있지만 이 프로젝트의 질문에 아무것도 안 보탠다.
- DB-bound 워크로드 — 커넥션 풀이 병목. 재밌지만 다른 실험이다.

**되돌릴 조건.** 캐시 실험이 끝나고 다른 워크로드와 비교하고 싶어지면. 그때는
**같은 리그에 섞지 말고 별도 프로젝트로.**

## 2. 물리 구성 — 1대 → 2대

**결정.** APC(192.168.55.9) 부하 생성 / BPC(192.168.55.164) SUT. **2대.**

**근거.** 이 실험의 주 지표는 **히트율(카운터)**이라 CPU 경합에 안 흔들린다.
그래서 원리적으로는 1대로도 결과가 나온다 — LLC 오염도 무관하고(앱 캐시는 GB,
L3는 MB), 네트워크 충실도도 무의미하다(LB hop 0.1ms vs 미스 페널티 ms 단위).

그럼에도 2대인 이유는 **지연을 같이 재기 때문**이다. 부하 생성기가 SUT 와 같은
머신에 있으면 부하가 커질수록 생성기가 CPU 를 뺏어서, 측정된 "포화 지점"이
실은 **생성기의 포화 지점**이 된다. 그리고 우리는 포화점 근처를 잰다(Phase 4).

**버린 것.** 3대 구성(부하 / SUT / DB 분리). origin 이 실 DB 가 아니게 되면서(#8)
DB 전용 머신이 필요 없어졌다.

**되돌릴 조건.** origin 을 실 DB 로 바꾸면 3대로 돌아가야 한다.

---

## 3. arm C(키 해시)를 추가 — 이게 없으면 틀린 교훈이 남는다

**결정.** A(자) / B(라운드로빈) / C(키 해시) 3 arm.

**근거.** A/B 만 하면 결론이 **"스케일아웃 하지 마라"**가 되는데, 그건 틀렸다.
진짜 원인은 스케일아웃이 아니라 **라운드로빈 + 로컬 캐시의 조합**이고,
해결책은 스케일업이 아니라 **라우팅**이다. C 가 그걸 증명한다.

C 를 넣으면 교훈이 **"로컬 캐시를 쓰면서 라운드로빈 하지 마라"**로 바뀐다 —
이게 맞는 교훈이다.

**되돌릴 조건.** 없음. C 는 이 프로젝트의 펀치라인이다.

---

## 4. A 는 경쟁자가 아니라 자(ruler)

**결정.** A(1 노드 × 캐시 N)는 비교 대상이 아니라 **기준선**이다.

**근거.** 히트율 71% 가 좋은 건지 나쁜 건지는 **"쪼개지 않았으면 몇 % 였나"**를
알아야 말할 수 있다. 92% 였다면 21%p 를 잃은 거고 73% 였다면 애초에 함정이 없던
워크로드다 — 완전히 다른 결론인데 A 가 없으면 구분이 안 된다.
**A 가 없으면 함정의 크기를 측정할 수 없다.** 함정이 주제인데.

**되돌릴 조건.** 없음.

---

## 5. nginx 를 A 에도 물린다

**결정.** 노드가 하나뿐인 A 도 nginx 를 거친다.

**근거.** A 만 nginx 를 건너뛰면 네트워크 홉 하나를 공짜로 아끼게 되어,
측정된 차이가 **"캐시 효과"인지 "LB 비용"인지 분리가 안 된다.**
홉을 상수로 고정해야 배치만 변수로 남는다.

## 6. 개인화 축 — 폐기

**결정.** 응답의 개인화 비율을 축으로 두지 **않는다.** 워크로드는 익명 객체 조회다.

**근거 (둘).**
1. **아무도 안 하는 설계를 재는 것에 가깝다.** 업계는 개인화를 캐싱하지 않고
   **캐시 경로 밖으로 뺀다** — fragment 캐싱(ESI) · 클라이언트 사이드 개인화 ·
   세그먼트 단위 키(등급 5개면 엔트리 5개) · **재료를 캐싱하고 조립은 매번**.
   마지막 게 우리 캐시다 — `product:42` 는 누구에게나 같아서 개인화가 들어올 자리가 없다.
2. **답이 계산으로 나온다.** 요청의 x% 가 개인화라 캐시가 안 되면
   히트율 = `(1-x) × 기본히트율`. **직선이다.**

**기준: 유도할 수 있는 건 유도하고, 유도 못 하는 것만 측정한다.**
skew 교차점은 LRU 축출 동역학과 부하 쏠림이 얽혀 유도가 안 된다 → 측정할 값어치가 있다.
개인화 축은 아니다.

**부수 정리.** "개인화된 캐시"는 대체로 존재하지 않는다. 개인화된 데이터(장바구니·세션)는
**캐시가 아니라 스토어**다. 경계선: **버려도 다시 만들 수 있으면 캐시, 못 만들면 저장소.**
`product:42` 는 DB 에서 다시 읽으면 되지만 장바구니는 날아가면 끝이다.

## 7. 게이트웨이 캐싱 arm — 폐기 (문서로만)

**결정.** nginx `proxy_cache` arm 을 런 매트릭스에 넣지 **않는다.** `TECHNICAL.md` 에 서술로 남긴다.

**근거.** 두 구성 모두 결과가 **연역된다**:
- nginx 1대 + 캐시 1개 → A 의 천장을 회복. **당연하다** — 안 쪼갰으니까.
- nginx 2대 + 각자 로컬 캐시 → **함정이 그대로 재현**. **당연하다** — B 와 같은
  메커니즘이 한 층 위에서 도는 것이다.

**연역되는 건 실험이 아니라 시연이다.** 시연은 문서에 들어간다.

**부수 정리 (중요).** 이 재귀가 이 프로젝트의 일반화다 — **함정은 "게이트웨이냐 앱이냐"가
아니라 "로컬 캐시를 가진 티어를 복제하는가"의 문제다.** 게이트웨이 캐싱이 안전해 보이는 건
원리적 면역이 아니라 **게이트웨이를 덜 스케일아웃하기 때문**이다. CDN 이 엣지마다 같은 파일을
중복 보관하는 것도 같은 구조인데, 거기선 중복이 **의도된 설계**다.

**그리고 게이트웨이 캐싱은 이커머스에서 벽에 부딪힌다.** `proxy_cache` 는 HTTP **응답** 캐시라
모든 사용자에게 같은 응답일 때만 성립한다. 헤더의 장바구니 개수 하나 때문에 캐시 키에
사용자를 넣어야 하고 → 엔트리가 사용자 수만큼 폭발 → 히트율 붕괴. **개인화 한 조각이
페이지 전체의 캐시 가능성을 죽인다.** 무효화도 약하다 — OSS nginx 엔 퍼지가 없어서
(NGINX Plus 나 서드파티 모듈 필요) 사실상 TTL 만료에 의존한다.

**되돌릴 조건.** "층을 올려도 못 도망친다"를 **실측으로** 보여주고 싶어지면
`nginx 2대 + 각자 proxy_cache + 앞단 RR` 로 넣는다. 결론을 강화하지 바꾸진 않는다.

## 8. origin = CSV 인메모리 + 고정 지연 (H2 아님)

**결정.** 실 DB 를 쓰지 않는다. CSV 를 통째로 메모리에 올려두고 고정 지연 후 반환.

**근거.** DB 는 단순히 노이즈가 아니라 **효과를 체계적으로 상쇄한다.**
DB 는 자기 버퍼 캐시를 갖는데 **그 캐시가 arm 마다 다르게 데워진다** —
B 는 미스가 많으니 DB 캐시를 더 넓게 데우고 → **B 의 미스가 A 의 미스보다 싸진다**
→ 우리가 재려는 "B 의 손해"를 DB 가 몰래 깎아준다.
**노이즈면 평균으로 없어지지만 이건 편향이라 안 없어지고, 하필 방향이 결론을 약하게 만드는 쪽이다.**

**미스 페널티는 파라미터여야지 측정 대상이 아니다.**

**되돌릴 조건.** "실 DB 를 붙이면 뭐가 달라지나" 자체가 질문이 되면. 그때는 이 편향을
알고 설계해야 한다(예: DB 캐시 끄기, 또는 arm 별 DB 인스턴스 분리).

## 9. 캐시 = 직접 짠 LRU (Caffeine 아님)

**결정.** `LinkedHashMap(accessOrder=true)` + `removeEldestEntry` + 락. 정확한 엔트리 수.

**근거.** Caffeine 을 안 쓰는 이유는 **무거워서가 아니다** — Caffeine 은 가볍고 빠르다.
1. **Caffeine 은 LRU 가 아니다.** W-TinyLFU(빈도 추정 기반 admission)를 쓴다 —
   **Zipf 쏠림에서 LRU 보다 잘하도록 설계된** 정책인데 하필 우리 축이 Zipf 쏠림이다.
   재는 대상이 바뀐다.
2. **크기가 근사치다.** `maximumSize(n)` 은 비동기 축출·세그먼트 구조 때문에 정확하지 않다.
   최상위 불변식이 "총 캐시 용량 고정"인데 근사면 곤란하다.
3. **분석이 안 된다.** ← 결정타. LRU+Zipf 는 이론 히트율을 근사할 수 있어서
   **"이론값 vs 실측값" 대조가 곧 기준소스**가 된다. W-TinyLFU 엔 닫힌 형태가 없다.

**부수 정리.** **중복 현상 자체는 정책 무관**이다 — 라운드로빈이면 두 노드가 같은 분포를
보므로 어떤 정책이든 같은 핫셋으로 수렴한다. 바뀌는 건 곡선의 모양과 해석 가능성이다.

> ⚠️ **함정**: `accessOrder=true` 면 **`get()` 이 자료구조를 변경한다**(엔트리를 뒤로 이동).
> **동시에 읽기만 해도 깨진다.** 예외가 뜨면 다행이고, 조용히 카운터가 유실되면
> 그럴듯하고 틀린 히트율이 나온다. 락 필수.

## 10. 캐시 크기 = 엔트리 개수 (바이트 아님)

**결정.** "캐시는 5,000 엔트리를 보관한다". 바이트로 재지 않는다.

**근거.** 바이트로 재면 런타임마다 "N개 = 몇 바이트"가 다르고(객체 헤더·정렬),
JVM 이면 힙+GC 여유+메타스페이스가 컨테이너 리밋을 넘겨 OOMKill 이 난다.
결국 **arm 간 "같은 캐시 크기"를 보장할 수 없다 → 최상위 불변식이 깨진다.**
엔트리로 재면 정확하고 언어 무관하며, 축(키공간÷캐시)이 무차원 비율이 된다.

**대가.** **도커 메모리 리밋이 메커니즘이 아니라 안전망으로 격하된다.**
메모리 압박 자체(GC·스왑·OOM)는 이 실험의 대상이 아니게 된다. 그건 다른 실험이다.

## 11. 스택 = Kotlin (JIT 우려는 기각)

**결정.** Kotlin + Spring Boot + Java 21.

**근거.** 앱이 하는 일이 HTTP 핸들러 + LRU + 카운터뿐이라 **언어에서 배울 게 없다.**
그러면 익숙한 쪽이 낫다 — 언어와 씨름하는 시간은 순손실이다.

**기각된 반대 논거(내가 냈다가 철회).** "JVM 은 JIT 워밍업이 겹쳐서 교란된다" →
**계산해보니 틀렸다.** 캐시 워밍업이 10⁵~10⁶ 요청 규모인데 C2 컴파일 임계치는 10⁴ 규모라
**긴 쪽이 짧은 쪽을 덮는다.** 게다가 주 지표인 히트율은 JIT 의 영향을 아예 안 받는다
(JIT 는 속도를 바꾸지 캐시에 어떤 키가 들었는지를 바꾸지 않는다).

**버린 것.** Go.

## 12. 축 재정의 — "워킹셋" → "키 공간"

**결정.** 축 이름은 **키 공간 ÷ 캐시**. "워킹셋"이라는 말을 쓰지 않는다.

**근거 (사용자 지적).** **워킹셋은 Denning 의 페이징 모델 용어**로 "최근 τ 시간 안에
참조된 페이지 집합"이다. 그 개념을 쓴다는 건 **집합 단위 지역성**을 전제하는 건데
우리는 **키 하나하나에 LRU** 를 건다. 세트 개념이 없다.

더 결정적으로 — **우리 합성 벤치엔 휴면 키가 없다.** Zipf 꼬리가 얇을 뿐 N 개 전부가
분포에 들어있어서 "활성 부분집합"이 존재하지 않는다. 그건 워킹셋이 아니라 **키 공간 N** 이다.

## 13. 캐시를 고정하고 키 공간을 스윕한다 (그 반대 아님)

**결정.** `CACHE_TOTAL = 10,000` 고정. 키 공간 N ∈ {5k, 10k, 20k, 40k, 80k} 스윕.

**근거 (사용자 논거 채택 — 내 안을 기각).**
내 안은 "키 공간 고정 + 캐시 스윕"이었고 근거는 "N 을 바꾸면 Zipf 분포가 바뀌어 축이
엉킨다"였다. **약한 논거였다.** N 이 커지는 건 교란이 아니라 **독립변수 그 자체**다 —
세상의 데이터가 늘어나는 걸 표현하는 게 축의 목적이다. 캐시를 줄여 그걸 흉내내는 건 **대리물**이다.

그리고 사용자 안엔 내가 못 본 이점이 있다: **캐시를 고정하면 실험 장치가 스윕 내내
고정된다** — A 는 언제나 정확히 10,000, B/C 는 언제나 정확히 5,000씩. 측정할 때마다
장치를 다시 만들지 않는다. 내 안은 매 지점마다 **arm 을 구분짓는 바로 그 값**을 바꾸는 것이었다.

**보너스.** 비율 0.5(N=5,000)가 **공짜 음성 대조군**이 된다 — 반쪽 캐시(5,000)에도
키 공간 전체가 들어가므로 **A·B·C 전부 히트율 100%, 손실 0 이어야 한다.**
여기서 차이가 나면 그건 발견이 아니라 **리그가 고장난 것**이다.

**남은 주의.** **비율은 완전한 무차원 축이 아니다.** 균등에선 히트율 = C/N 이라 비율만으로
결정되지만, Zipf 에선 히트율 ≈ ln(C)/ln(N) 이라 C 와 N 이 각각 영향을 준다.
캐시를 고정했으니 실질적으로는 문제가 안 되지만, **결과를 "비율의 함수"로 일반화할 때 조심.**

---

## 14. modulo 해싱 (consistent 아님)

**결정.** arm C 는 `hash $uri` — `consistent` 없이.

**근거.** consistent hashing 은 **노드 수가 변할 때**의 재배치 폭풍(2→3대면 키의 2/3 가
이사 → 미스 폭풍)을 막는 도구인데, 우리는 실행 중 노드 수를 바꾸지 않는다.
붙이면 vnode 때문에 분할이 정확히 반반이 아니게 되어 **노이즈만 생긴다.**

**되돌릴 조건.** 노드 수를 런 중에 바꾸는 실험(탄력성)을 하면 그때는 consistent 가 주인공이다.

## 15. RPS 를 히트율 지도에서 분리

**결정.** 히트율 지도는 RPS 하나로. 부하 스윕은 별도 페이즈.

**근거.** **히트율은 RPS 에 (거의) 무관하다** — 접근 시퀀스와 캐시 크기의 함수지
초당 몇 개 쐈느냐의 함수가 아니다. 곱하면 같은 숫자를 4번 재는 것이고
런이 180 → 70 으로 준다.

**부수 이득.** **"히트율이 RPS 무관"이 자체로 검증 가능한 주장**이다 — 같은 지점을
두 RPS 로 돌려 히트율이 같은지 보면 된다. 다르면 카운터 경합이다. **공짜 검증이 하나 더.**

## 16. 사전 등록 — 예측을 실측 전에 커밋한다

**결정.** 페이즈마다 `spec.md`(질문·수학적 예측·판정 기준)를 **런 전에 커밋**하고,
그 다음에 `result.md`(실측) → `gate.md`(예측 대비 판정).

**근거.** 숫자를 본 뒤에 예측을 쓰면 그건 **예측이 아니라 사후 합리화**고,
무슨 결과가 나오든 항상 맞아떨어진다. **항상 맞는 이론은 아무 말도 안 하는 이론이다.**
**커밋 타임스탬프가 증거다** — "이 예측은 결과를 보기 전에 쓰였다"가 git 에 박힌다.

하네스의 **절단 계약**("테스트 설계는 구현 diff 를 보기 전에 spec 에 먼저 쓴다")과
같은 원리다. 답이 질문을 오염시키지 못하게 하는 것 — 실험에선 그게 **사전 등록**이다.

## 17. 스레드 통제 — 논쟁하지 않고 측정해서 정한다

**상태.** ❓ **Phase 0 의 산출물로 결정한다.**

**쟁점.** Spring Boot 기본 Tomcat 200 스레드면 A 는 200, B/C 는 2노드×200 = **400**.
**동시성 capacity 가 arm 마다 다르다.** 미스는 origin 에서 블로킹하며 스레드를 잡으므로
미스가 많은 B 가 더 쓴다.

**사용자 의견.** "도커 컴포즈로 CPU 를 제한하는 게 가장 깔끔하다."
→ **주 통제기로는 맞다. 다만 CPU 리밋은 블로킹 스레드를 못 막는다** —
origin 에서 5ms 블로킹하는 스레드는 CPU 를 안 쓴다.

**해소 방법.** 계산상 CPU 한계(~2~5만 RPS)와 스레드 한계(200스레드/5ms = 4만 RPS)가
**비슷해서 어느 쪽이 먼저 묶일지 지금은 모른다.** Phase 0 에서 **CPU 사용률과 활성
스레드 수를 같이 계측**해서 정한다:
- CPU 100% + 스레드 널널 → **CPU 가 묶는다** → 사용자 말대로 CPU 리밋만으로 충분
- CPU 60% + 스레드 200 붙음 → **스레드가 묶는다** → 총량 고정 필요 (A:200 / B·C:100씩)

## 18. 워밍업 — 고정 횟수가 아니라 plateau 탐지

**결정.** "20 × 키공간 = 200,000 요청" 같은 고정 숫자를 쓰지 않는다.
`/admin/stats` 를 주기적으로 긁어 **히트율 기울기가 0 에 수렴하면** 워밍업 종료.

**근거.** 고정 횟수는 내 짐작이었다. N=80,000 일 때와 5,000 일 때 필요한 워밍업이
다른데 한 숫자로 못 박으면 **한쪽은 낭비고 한쪽은 오염**이다.
"수학적 계산과 실제 결과를 비교한다"는 원칙을 워밍업에도 적용한 것.
