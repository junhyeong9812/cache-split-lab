## 0. 작업 기준

| 칸 | 내용 |
|----|------|
| **목표** | 인메모리 캐시를 가진 티어를 복제할 때 히트율이 얼마나 무너지는지, 그게 **접근 분포와 캐시 여유의 함수로 어떻게 변하는지** 지도를 그린다. 그리고 **B(라운드로빈)와 C(키 해시)의 우열이 뒤집히는 교차점**을 찾는다 |
| **불변식** | ① **총 캐시 용량 = arm 무관 동일** (A: 10,000 / B·C: 5,000×2) ② **총 CPU·총 스레드 동일** ③ arm 간 차이는 **배치뿐** — 앱 jar·origin·k6 시나리오·요청 시퀀스(동일 seed)·nginx 홉 전부 상수 ④ arm C 의 키→노드 매핑은 **결정적** |
| **기준소스** | **① 이론 히트율** (LRU+Zipf 모델 — ❗미확정, 빚 A) **② 음성 대조군** (비율 0.5 → 손실 0 이어야 함) **③ 교차 검증** (앱 miss 합 ≡ origin 요청 수, 앱 카운터 ≡ k6 집계, nginx `$upstream_addr` ≡ 앱 `nodeId`) |
| **금지영역** | `docs/study/` (사용자 학습 노트) · 이미 커밋된 `spec.md` 의 예측 (사후 수정 = 사전 등록 무효) |
| **검증** | 페이즈마다 `gate.md` — 예측 대비 판정 + sanity 체크(`ARCHITECTURE.md §6`) |
| **stakes** | **중간.** blast radius 는 작다(로컬·불가역 없음·재실행 가능). 그럼에도 낮음이 아닌 이유: **지배적 실패모드가 "조용히 틀린 숫자"**다. 히트율은 맞아도 71%, 틀려도 71%처럼 보인다. 카운터 경합 하나, Zipf 아닌 Zipf 생성기 하나면 확신에 찬 거짓 결론이 나오고 — **틀린 교훈을 배우는 건 아무것도 안 배우는 것보다 나쁘다.** 손실이 기계가 아니라 **멘탈 모델**에 간다 |

## 1. 사전 등록 규칙 (이 프로젝트의 헌법)

**페이즈마다 3파일, 이 순서로:**

| 파일 | 내용 | 시점 |
|------|------|------|
| `spec.md` | 질문 · **수학적 예측(유도 과정 포함)** · 측정 방법 · **판정 기준** | ★ **런 전에 커밋** |
| `result.md` | 실측 raw — 히트율 · 노드별 분포 · 지연 · 에러율 · 환경 | 런 후 |
| `gate.md` | **예측 대비 판정** · 해석 · 다음 페이즈 입력 | 판정 시 |

**규칙:**
1. **`spec.md` 를 커밋하기 전에는 런을 돌리지 않는다.** 커밋 타임스탬프가
   "이 예측은 결과를 보기 전에 쓰였다"의 유일한 증거다.
2. **커밋된 예측은 고치지 않는다.** 틀렸으면 틀린 채로 두고 `gate.md` 에서
   왜 틀렸는지 쓴다. **빗나간 예측이 맞은 예측보다 정보량이 많다.**
3. 판정 기준(예: "손실이 예측 ±5%p 안이면 일치")도 `spec.md` 에 **미리** 적는다.
   결과를 보고 기준을 정하면 기준이 아니다.

> 숫자를 본 뒤에 예측을 쓰면 그건 사후 합리화고, 무슨 결과가 나오든 항상 맞아떨어진다.
> **항상 맞는 이론은 아무 말도 안 하는 이론이다.**

## 2. 페이즈 분해

| 페이즈 | 폴더 | 목표 | 선행 | 런 | 완료 조건 |
|--------|------|------|------|-----|----------|
| **0** | `phase-0-saturation/` | **세 arm 각각의 포화점** + 무엇이 묶이는가(CPU/스레드) | 구현 | 램프업 3회 | RPS 기준선 확정 · 스레드 통제 여부 확정 · **A/B/C 포화점 차이 기록** |
| **1** | `phase-1-uniform/` | s=0 (균등) — 비율 5 × arm 3 | 0 | 15 | 음성 대조군(비율 0.5) 통과 + 예측 대비 판정 |
| **2** | `phase-2-zipf1/` | s=1 (고전 Zipf) | 1 | 15 | 예측 대비 판정 |
| **3** | `phase-3-zipf2/` | s=2 (강한 쏠림) | 2 | 15 | 예측 대비 판정 · **B/C 교차점 존재 여부** |
| **4** | `phase-4-load/` | 부하 구간별 트레이드오프 — RPS 80/90/100/110% × arm 3 | 1~3 | 12~24 | 구간별 우열 지도 |

**Phase 1 을 먼저 하는 이유.** 예측 효과가 제일 크다(50%p). **리그가 고장났으면
50%p 예측이 빗나가는 건 못 놓친다.** 작은 효과(Phase 3, ~0%p)부터 시작하면
"안 나온 건지 원래 없는 건지" 구분이 안 된다. **큰 신호로 장비를 먼저 검증한다.**

### 축

- **비율** (키 공간 ÷ 캐시): 0.5 / 1 / 2 / 4 / 8 → N = 5k / 10k / 20k / 40k / 80k
- **캐시**: `CACHE_TOTAL = 10,000` **고정** (A: 10,000 / B·C: 5,000씩)
- **skew**: s = 0 / 1 / 2 (페이즈 경계)
- **arm**: A(자) / B(라운드로빈) / C(키 해시) — **세 arm 전부, 모든 페이즈에서**

## 3. Phase 0 — 세 arm 전부 잰다

**초안이 틀렸었다.** "A 기준 포화점"으로 잡았는데, **B/C 는 A 보다 먼저 포화할 가능성이
크다** — 미스가 많으니 origin 호출이 많고 스레드가 더 쌓인다. A 의 100% 가 B 에겐
140% 일 수 있고, 그러면 "80~110% 구간"을 잰다면서 실은 **B 는 이미 다 터진 상태**를 잰다.

**그리고 그 포화점 차이 자체가 결과다.** "B 가 A 보다 30% 먼저 무릎 꿇는다" —
이게 **히트율 손실이 용량 손실로 번역된 숫자**고, 실무에서 진짜 아픈 지점이다.
히트율 7%p 보다 이게 더 와닿는다.

Phase 0 산출물:
- A / B / C 각각의 포화 RPS
- **셋의 차이** ← 결과
- 무엇이 묶는가 (CPU 사용률 vs 활성 스레드 수) → 빚 B 해소
- Phase 4 의 100% 기준선 정의 (셋 중 무엇을 기준으로 할지 포함)

## 4. 페이즈 간 의존

```
구현 ──► Phase 0 ──┬──► Phase 1 (s=0) ──► Phase 2 (s=1) ──► Phase 3 (s=2)
   빚 A 해소        │        │                                      │
   (이론 확정)      │        └──── 흥미로운 지점 ─────┬─────────────┘
                    │                                 ▼
                    └─────────── RPS 기준선 ──────► Phase 4 (부하 구간)
```

- **빚 A(이론 확정)는 Phase 1 spec 의 선행이다.** 근거 없는 예측을 사전 등록하면
  빗나갔을 때 "이론이 틀렸나 내가 틀렸나"를 구분할 수 없다.
- Phase 4 는 Phase 1~3 이 고른 지점 + Phase 0 의 RPS 기준선을 입력으로 받는다.

## 5. 전 페이즈 공통 sanity (`ARCHITECTURE.md §6`)

매 런 자동 검사. **하나라도 깨지면 그 런의 숫자는 버린다.**

- [ ] 음성 대조군 — 비율 0.5 에서 A·B·C 전부 히트율 ≈ 100%, 손실 ≈ 0
- [ ] 앱 miss 합 ≡ origin 요청 수
- [ ] 앱 히트율 ≡ k6 가 본 `cached` 비율
- [ ] arm C — 두 노드 캐시 키 **교집합 ≈ 0**
- [ ] arm B — 두 노드 캐시 키 **교집합 ≈ 100%** (고 skew 에서. 균등에선 다르다 → Phase 1 에서 확인)
- [ ] arm C — 같은 키 반복 요청 시 항상 같은 `nodeId`
- [ ] 캐시 상주 수 ≡ 설정한 capacity (정확히)
- [ ] Zipf 생성기가 실제로 Zipf (시퀀스 분포 직접 검정)
- [ ] 에러율 0 (아니면 그 구간 **지연 지표 무효** — 느린 요청이 에러로 빠져 p99 가 거짓으로 좋아진다)
