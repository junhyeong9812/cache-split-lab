# GLOSSARY — cache-split-lab

> 이 프로젝트 문서에 나오는 용어를 처음 보는 사람 기준으로 설명한다.
> 순서는 "알아야 다음 걸 이해할 수 있는" 순서다. 위에서부터 읽으면 된다.

---

## 1. 캐시 기초

### 캐시 (cache)
느린 곳에서 가져온 걸 빠른 곳에 **베껴 두고 재사용**하는 것. 우리 실험에서는
CSV 원본(`origin`)에서 읽은 값을 앱의 메모리에 들고 있는 것.

캐시의 조건은 넷이다 — 넷이 다 맞아야 캐시가 이득이다:

1. **읽기 ≫ 쓰기** (자주 읽고 드물게 바뀜)
2. **접근이 쏠려 있음** (일부만 자주 읽힘 → §4 Zipf)
3. **다시 만드는 비용 ≫ 꺼내는 비용** (원본이 느림)
4. **조금 낡아도 됨** (staleness 허용)

4번이 0이면(한 순간도 낡으면 안 됨) 캐시는 애초에 불가능하다.

### 히트 / 미스 (hit / miss)
- **히트**: 찾는 값이 캐시에 있었다 → 빠르게 반환
- **미스**: 캐시에 없었다 → 원본까지 가서 가져온 뒤 캐시에 넣고 반환

### 히트율 (hit rate)
전체 요청 중 히트의 비율. `히트 / (히트 + 미스)`.
**이 프로젝트의 주 지표.** 히트율 90%는 요청 10개 중 9개가 캐시에서 나왔다는 뜻.

### 미스 페널티 (miss penalty)
미스 한 번이 추가로 무는 비용(시간). 원본까지 다녀오는 시간.
**우리 실험에서 이건 측정 대상이 아니라 우리가 정하는 값(파라미터)이다.**
그래서 `origin`을 실제 DB가 아니라 "고정 지연 후 반환" 스텁으로 둔다 —
미스 비용이 항상 정확히 같아야 arm 간 차이가 순수하게 **미스 횟수** 차이로만 남는다.

### 워킹셋 (working set)
**실제로 접근되는 서로 다른 키의 집합.** 크기로 말한다.

상품이 100만 개 있어도 오늘 조회되는 건 4만 개뿐이라면 워킹셋은 4만이다.
캐시는 DB 전체가 아니라 **워킹셋만 담으면** 충분하다.

> **워킹셋 ÷ 캐시 크기**가 이 실험의 축 하나다. 이 비율이
> - ≪ 1 (캐시가 워킹셋보다 큼) → 다 담긴다 → 히트율 100% → 쪼개도 티 안 남
> - ≫ 1 (캐시가 한참 작음) → 거의 다 미스 → 히트율 ~0% → 쪼개도 티 안 남
> - **≈ 1 부근** → **여기서만 함정이 터진다**

## 2. LRU — 뭘 버릴 것인가

**LRU = Least Recently Used = "가장 오래 안 쓴 것을 버린다".**

캐시는 유한하다. 꽉 찬 상태에서 새 값을 넣으려면 **뭔가를 버려야 한다**(축출/eviction).
뭘 버릴지 정하는 규칙을 **축출 정책**이라 하고, LRU는 그중 가장 흔한 것이다.

논리: *최근에 쓴 건 또 쓸 가능성이 높다*(시간적 지역성). 그러니 **제일 오래 방치된 걸 버린다.**

### 동작 (캐시 3칸)

```
요청:   A       B        C          A            D
       ─────   ─────    ───────    ─────────    ─────────────
캐시:  [A]     [B A]    [C B A]    [A C B]      [D A C]
                                     ↑             ↑
                              A를 다시 씀       꽉 참 →
                              → 맨 앞으로       제일 오래 안 쓴 B 축출

(왼쪽 = 최근에 쓴 것, 오른쪽 = 오래된 것)
```

### 다른 정책들

| 정책 | 버리는 기준 |
|------|------------|
| **LRU** | 가장 오래 **안 쓴** 것 ← 우리가 쓰는 것 |
| FIFO | 가장 **먼저 들어온** 것 (쓰든 말든) |
| LFU | 가장 **적게 쓰인** 것 (횟수 기준) |
| Random | 아무거나 |

### 왜 우리는 LRU를 직접 짜는가

Kotlin 진영의 표준 캐시 라이브러리 **Caffeine 은 LRU 가 아니다** — **W-TinyLFU**라는
다른 정책(빈도를 추정해서 "들여보낼지 말지"를 결정하는 admission policy)을 쓴다.
그건 **Zipf 쏠림 workload 에서 LRU 보다 잘하도록 특별히 설계된** 물건인데,
하필 우리 실험의 축이 **Zipf 쏠림**이다. 그걸 쓰면 재는 대상이 바뀐다.

직접 짜는 이유 셋:

1. **크기가 정확하다.** Caffeine 의 `maximumSize(n)` 은 근사치다(비동기 축출 · 세그먼트 구조).
   우리 최상위 불변식이 "총 캐시 용량 고정"이라 근사면 곤란하다.
2. **동작이 보인다.** 숫자가 이상할 때 40줄을 읽으면 이유를 안다.
3. **검증이 된다.** LRU + Zipf 는 **이론 히트율을 계산할 수 있다.** W-TinyLFU 는 못 한다.
   → "이론값 대비 실측값"이 우리의 검증 기준이 된다.

> ⚠️ **Kotlin 함정**: `LinkedHashMap(capacity, loadFactor, accessOrder = true)` +
> `removeEldestEntry` 오버라이드 = LRU 가 공짜로 나온다. 그런데 `accessOrder = true` 면
> **`get()` 이 자료구조를 변경한다**(엔트리를 맨 뒤로 옮김). 즉 **동시에 읽기만 해도 깨진다.**
> 예외가 뜨면 다행이고, 조용히 카운터가 유실되면 **그럴듯하고 틀린 히트율**이 나온다.
> 반드시 락으로 감싼다.

## 3. arm — 실험군

**arm = 나란히 놓고 비교하는 하나의 구성.**

임상시험 용어다. 신약 시험에서 "약 먹는 그룹"과 "가짜약 먹는 그룹"을 각각 arm 이라 부른다.
"실험군"이라고 읽으면 된다.

우리는 셋을 비교한다:

| arm | 폴더 | 구성 | 역할 |
|-----|------|------|------|
| **A** | `arms/a-single` | 1 노드 × 캐시 N | **자(ruler)** — 안 쪼갰을 때의 천장 |
| **B** | `arms/b-roundrobin` | 2 노드 × 캐시 N/2, 라운드로빈 | 함정 |
| **C** | `arms/c-keyhash` | 2 노드 × 캐시 N/2, 키 해시 | 해소 + 대가 |

**총 캐시 용량은 셋 다 같다** (A 는 N 하나, B/C 는 N/2 를 둘). 같은 자원을 어떻게
배치하느냐만 다르다 — 이래야 "같은 돈으로 뭐가 나은가"가 성립한다.

**A 는 네 번째 경쟁자가 아니라 자다.** 히트율 71% 가 좋은 건지 나쁜 건지는
"안 쪼갰으면 몇 % 였나"를 알아야 말할 수 있다.

## 4. Zipf — 얼마나 쏠려 있는가

**Zipf 분포 = "소수가 대부분을 먹는" 쏠림 분포.**

언어학자 George Zipf 가 단어 빈도에서 발견했다 — 1등 단어가 2등의 2배, 3등의 3배로 나온다.
즉 **빈도 ∝ 1 / 순위**. 상품 조회수, 도시 인구, 웹 트래픽, 유튜브 조회수가 다 이 모양이다.

쏠림 정도는 지수 `s` 로 조절한다: **빈도 ∝ 1 / 순위^s**

### 상품 1,000개일 때 상위 10개가 먹는 트래픽 비중

| `s` | 성격 | 상위 10개 몫 | 현실 비유 |
|-----|------|-------------|----------|
| **0** | 완전 평평 — 모든 상품이 똑같이 인기 | **1%** | 롱테일 카탈로그 탐색 |
| **1** | 고전적 Zipf | **약 39%** | 평상시 이커머스 |
| **2+** | 극단적 쏠림 — 몇 개가 다 먹음 | 대부분 | **플래시 세일** |

### 왜 이게 이 프로젝트의 핵심 축인가

**`s` 하나가 "중복 캐시는 낭비인가 기능인가"를 뒤집는다.**

- **`s` 낮음(평평)** → 뜨거운 게 따로 없어서 캐시를 쪼갠 손해가 크다.
  각 노드가 워킹셋의 절반밖에 못 담고, 나머지 절반은 계속 미스 → **중복 = 낭비**
- **`s` 높음(쏠림)** → 뜨거운 몇 개는 **작은 캐시에도 어차피 다 들어간다.**
  쪼개도 손해가 없다. 오히려 모든 노드가 그 키를 갖고 있어야 아무 노드나 응답할 수
  있으니 → **중복 = 기능** (CDN 이 엣지마다 같은 파일을 두는 것과 같은 이유)

그리고 **같은 서비스가 이벤트 시작 순간 `s` 축의 왼쪽 끝에서 오른쪽 끝으로 순간이동한다.**
평상시 카탈로그 탐색(s≈1) → 플래시 세일 시작(s≈3). 그래서 그렇게 자주 터진다.

### seed
난수 생성기의 시작값. **같은 seed → 같은 요청 순서.**
arm A/B/C 에 **완전히 동일한 요청 시퀀스**를 먹여야 비교가 성립하므로 seed 를 고정한다.

## 5. 로드밸런서와 라우팅

### 로드밸런서 (LB) / nginx
요청을 받아서 **뒤에 있는 여러 노드 중 하나로 넘겨주는** 중계자. 우리는 nginx 를 쓴다.
뒤에 있는 노드들을 **upstream** 이라 부른다.

> **nginx 는 arm A 에도 물린다.** A 는 노드가 하나라 분산할 게 없지만, A 만 nginx 를
> 건너뛰면 A 는 네트워크 홉 하나를 공짜로 아끼게 되어 "캐시 효과"와 "LB 비용"이 섞인다.
> 셋 다 지나게 해서 **홉을 상수로 고정**한다.

### 라운드로빈 (round-robin) — arm B
**순서대로 돌아가며** 배분. 1번, 2번, 1번, 2번… 요청 내용과 무관하게 번갈아 준다.

문제: `user:42` 를 물어보는 요청이 이번엔 1번 노드, 다음엔 2번 노드로 간다.
→ **두 노드가 각자 전체 키 공간을 본다** → 각자 LRU 로 "가장 뜨거운 절반"을 남긴다
→ 둘이 같은 분포를 보니 **둘 다 똑같은 키를 캐싱한다** → 유효 캐시 = N/2 (절반이 복제본)

### 키 해시 라우팅 (key hash) — arm C
**요청이 다루는 키로 목적지를 정한다.** `hash("user:42") % 2 = 0` → 항상 1번 노드.

→ 각 노드가 **자기 몫 키만** 본다 → 중복 없음 → 유효 캐시 = N (온전히 씀)

**대가**: 뜨거운 키 하나가 트래픽의 30% 를 먹으면 **그 키를 담당하는 노드 한 대가 30% 를
혼자 받는다.** 라운드로빈이면 고르게 흩어졌을 부하다. **`s` 가 높을수록 이 쏠림이 심해진다.**

> **이 대가가 §4 의 이득과 정반대로 움직인다.** s 가 낮으면 C 가 이기고,
> s 가 높으면 B 가 역전할 수 있다. **그 교차점을 찾는 게 이 프로젝트의 산출물이다.**

### modulo 해싱 vs consistent 해싱
`hash(key) % N` 이 가장 단순한 키 해시다. **노드 수 N 이 안 변하면 이걸로 충분하다.**

N 이 바뀌면(2대→3대) `%2` 가 `%3` 이 되면서 **거의 모든 키의 목적지가 바뀌고**
전 노드의 캐시가 통째로 무의미해진다(미스 폭풍). **consistent hashing** 은 해시 공간을
원(ring)으로 놓아 이동량을 ~1/N 로 줄인다. 실무가 이걸 쓰는 이유다.

**우리는 노드 수가 2로 고정이므로 modulo 로 충분하다.** consistent hashing 은
"실무에선 왜 이걸 쓰나"로 문서에만 남긴다.

## 6. 측정 용어

### p50 / p95 / p99 (백분위 지연)
- **p50** = 절반의 요청이 이보다 빨랐다 (중앙값)
- **p99** = 99% 의 요청이 이보다 빨랐다 → **가장 느린 1%** 가 얼마나 느린가

평균은 꼬리를 숨긴다. 평균 10ms 인데 p99 가 2초면 100명 중 1명은 2초를 기다린 것이다.
**사용자 체감은 평균이 아니라 꼬리에서 온다.**

### SUT (System Under Test)
측정 대상 시스템. 우리 경우 BPC(192.168.55.164) 위의 nginx + 앱 노드 + origin.

> 부하 생성기(k6)는 **반드시 SUT 와 다른 물리 머신**에 둔다. 같은 머신이면
> 부하가 커질수록 생성기가 CPU 를 뺏어서, 측정된 "포화 지점"이 실은
> **생성기의 포화 지점**이 된다.

### k6
부하 생성 도구. 정해진 도착률로 요청을 쏟아붓고 지연·처리량·에러율을 기록한다.
APC(192.168.55.9)에서 실행한다.

### 워밍업 (warmup)
측정 전에 캐시를 채워 **정상상태**로 만드는 구간. 빈 캐시로 시작하면 초반엔 전부 미스라
히트율이 낮게 나온다. 그 구간은 버리고 안정된 뒤부터 잰다.

우리 워밍업은 **캐시 기준**이다 — LRU 가 핫/콜드 순서를 정리할 만큼 요청이 돌아야 하므로
워킹셋을 여러 번 훑는 규모(10⁵~10⁶ 요청)가 필요하다. JVM 의 JIT 워밍업(10⁴ 규모)은
그보다 한 자릿수 짧아서 **캐시 워밍업에 흡수된다** — 따로 설계하지 않는다.
