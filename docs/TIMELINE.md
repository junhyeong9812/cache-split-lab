# TIMELINE — 무슨 일이 있었나

> 2026-07-17. 설계 대화부터 첫 실험까지의 기록.
> 무엇을 했고, 무엇이 틀렸고, 어떻게 알았고, 어떻게 고쳤는지를 순서대로 적는다.
> 결과는 표로, 분석은 산문으로.
>
> 이 문서의 목적은 자랑이 아니라 **재구성 가능성**이다. 여섯 달 뒤에 이 숫자가
> 어디서 왔는지 물었을 때 답할 수 있어야 하고, 특히 **틀렸던 것들이 언제 어떻게
> 드러났는지**가 남아야 한다.

## 1. 환경 — 그리고 첫 놀람

작업을 시작하면서 이 머신이 APC 인지 BPC 인지 확인했더니 **둘 다 아니었다.**

| 역할 | 호스트 | IP | CPU | 메모리 | 비고 |
|------|--------|-----|-----|--------|------|
| 개발 | `jun-Victus-...` | 192.168.55.114 | 24 | 36 GB | 여기서 조종만 한다 |
| 부하 (APC) | `jun-mini1` | 192.168.55.9 | 8 | 15 GB | k6 (docker 로) |
| SUT (BPC) | `jun` | 192.168.55.164 | 16 | 31 GB | nginx + 앱 + origin |

설계 문서에는 APC/BPC 2대 구성으로 적혀 있었는데 실제로는 **개발 노트북이 제3의
머신**이었다. 결과적으로 이게 더 낫다 — 조종 노드가 SUT 와 분리돼 있으면 러너의
폴링(`/admin/stats` 를 10초마다 긁는다)이 측정 대상의 CPU 를 안 먹는다.

SSH 는 기존 키로 양쪽 다 열려 있었다. 사용자가 대화에 붙여넣었던 키는 결국 쓰지
않았다(GitHub 는 별도 키, 이건 내부 리눅스 PC 전용이라고 확인됨).

**알려진 오염원 하나:** BPC 에 `ollama` 컨테이너가 2주째 healthy 상태로 돌고 있다
(포트 11434). 사용자의 작업이라 끄지 않았다. 유휴 상태면 CPU 를 안 쓰지만 실험
도중 누가 질의하면 16 코어를 먹어서 측정을 오염시킬 수 있다. 우리 포트
(80/8081/8082/9090)와 충돌은 없다. Phase 1 부터 런마다 BPC 의 load average 를
같이 기록해 감시한다.

## 2. 구현과 검증

### LruCache

`LinkedHashMap(accessOrder=true)` + `removeEldestEntry` + `ReentrantLock`.
단위 테스트 6건을 먼저 썼고 전부 통과했다.

| 테스트 | 검증하는 것 | 결과 |
|--------|------------|------|
| LRU 순서 | A B C A D → B 축출 (GLOSSARY §2 예시 그대로) | PASS (7ms) |
| get 이 접근순서 갱신 | 안 하면 FIFO 가 된다 | PASS (17ms) |
| capacity 초과 없음 | 10,000 회 삽입 중 매번 검사 | PASS (19ms) |
| 카운터 정확성 | 히트 2 / 미스 2 / 히트율 0.5 | PASS (18ms) |
| **동시 접근 카운터 유실** | 16스레드 × 5,000 = 80,000 요청 | **PASS (76ms) — 유실 0** |
| resetCounters | 캐시는 남기고 카운터만 0 | PASS (1ms) |

다섯 번째가 이 파일의 존재 이유다. `accessOrder=true` 는 **`get()` 이 자료구조를
변경**하므로 락 없이 동시 접근하면 깨지고, 카운터가 락 밖에 있으면 증가가 유실된다.
유실된 히트율은 **그럴듯하고 틀린 숫자**로 나온다 — 71% 가 맞아도 71%, 틀려도 71%
처럼 보인다. 그래서 여기서 잡아야 했다.

### Zipf 생성기

이론값(`H(k,s)/H(n,s)`)과 대조했다. N=1,000, 표본 200만.

| s | 상위 10개 실측 | 이론 | 오차 |
|---|--------------|------|------|
| 0.0 | 1.01% | 1.00% | 0.010%p |
| 0.5 | 8.16% | 8.12% | 0.035%p |
| **1.0** | **39.16%** | **39.13%** | **0.031%p** |
| 1.5 | 78.28% | 78.27% | 0.002%p |
| 2.0 | 94.27% | 94.27% | 0.003%p |

시드 재현성(같은 시드 → 같은 시퀀스, 다른 시드 → 다른 시퀀스)도 통과. s=0 의
균등성은 최대 편차 2.60%.

s=1 에서 39.16% 가 나온 게 반갑다 — `GLOSSARY.md` 에 "상품 1,000개면 상위 10개가
약 39%" 라고 써뒀던 그 숫자가 실측으로 확인됐다.

## 3. 문제 ① — origin 포트를 안 열었다

**증상:** 첫 스모크 테스트가 `RuntimeError: 헬스체크 타임아웃` 으로 죽었다.

**혼란:** 수동으로 compose 를 올려보니 세 컨테이너가 35초 만에 전부 healthy 였다.
그런데 러너는 120초를 기다리다 죽었다.

**진단:** 랩탑에서 BPC 의 포트를 하나씩 찔러봤다.

| 대상 | 랩탑 → BPC | BPC 내부 |
|------|-----------|---------|
| :8081 (app-1) | 200 | 200 |
| :80 (nginx) | 404 | — |
| **:9090 (origin)** | **실패** | **실패** |

`docker ps` 의 PORTS 열에서 origin 만 비어 있었다. **compose.yml 에 origin 의
`ports:` 를 안 썼다.** 앱은 도커 네트워크로 `origin:9090` 에 잘 닿으니 실험 자체는
돌았을 텐데, 러너가 밖에서 `/admin/stats` 를 못 긁었다.

**왜 중요한가:** origin 의 요청 수는 시스템 전체의 미스 횟수이고, **앱들의 miss 합과
일치해야 하는 교차 검증의 한쪽**이다. 이게 없으면 카운터 경합이나 몰래 일어난
재시도를 잡을 방법이 없다. 단순한 설정 누락이었지만 검증 장치를 통째로 날릴 뻔했다.

**교훈:** 헬스체크가 "타임아웃"이라고만 말해서 어디가 문제인지 몰랐다. 어느 대상이
안 되는지 찍어서 알려줬으면 1분 만에 끝났을 것이다.

## 4. 문제 ② — 워밍업이 수렴하지 않았다 (그리고 이건 진짜였다)

포트를 고치고 다시 돌리니 끝까지 갔다. 그런데 결과가 이상했다.

| 항목 | 값 |
|------|-----|
| 워밍업 plateau | **False** (80초 돌렸는데 미도달) |
| 워밍업 최종 구간 히트율 | 0.9204 (계속 오르는 중) |
| 측정 히트율 | 0.9615 (워밍업보다 **높다**) |
| **캐시 상주 / 용량** | **6,638 / 10,000** |
| 교차 검증 (miss ≡ origin) | True (632 ≡ 632) ✅ |

워밍업 곡선이 0.575 → 0.745 → 0.816 → 0.852 → 0.872 → 0.899 → 0.913 → 0.920 으로
**계속 오르기만 하고 평평해질 기미가 없었다.**

**진단은 캐시 상주 수가 줬다: 6,638 / 10,000.**

비율 1 에서는 캐시 용량(10,000) 이 키 공간(10,000) 과 같다. 그러면 **arm A 는 축출을
아예 안 한다.** 한 번 들어온 키는 영원히 남는다. 그래서 A 의 미스는 오직 "처음 보는
키"일 때만 발생하고, 히트율은 100% 에 **점근**한다 — 이건 캐시 문제가 아니라
**쿠폰 수집가 문제**다.

그리고 Zipf 분포에서는 이 수렴이 극도로 느리다. 꼬리의 차가운 키들이 좀처럼 안
뽑히기 때문이다. 500 RPS × 80초 = 40,000 요청을 쏟아부었는데도 10,000 개 키 중
6,638 개만 캐시에 들어왔다.

**이게 왜 심각한가.** 수렴 전에 측정하면 A 의 히트율이 과소평가된다. 그러면
손실(A − B)도 과소평가된다. 즉 **우리 가설을 약하게 만드는 방향의 편향**이다.
자기 가설에 유리한 편향은 눈에 잘 띄지만 불리한 편향은 "그냥 효과가 작았나 보다"
로 넘어가기 쉽다. 그게 더 위험하다.

**해결: 결정적 프리웜.** 측정 전에 모든 키를 정확히 한 번씩 훑는다
(`k6/prewarm.js`, k6 의 `shared-iterations` + `iterationInTest` 전역 인덱스).
nginx 를 통과하므로 arm 별 라우팅이 그대로 적용된다.

이건 치팅이 아니다. 우리가 재려는 건 **정상상태** 히트율이고, 프리웜은 거기 빨리
도달할 뿐 도착점을 바꾸지 않는다. 세 arm 에 똑같이 적용되고, 프리웜 후에도 Zipf
워밍업을 돌려 LRU 가 핫셋 순서를 정리하게 한다.

**효과:**

| | 프리웜 전 | 프리웜 후 |
|---|---------|----------|
| 캐시 적재 | 6,638 / 10,000 (80초 후) | **10,000 / 10,000 (11.7초)** |
| 워밍업 plateau | False (80초) | **True (40초)** |
| 측정 히트율 | 0.9615 (미수렴) | **1.0000** |

## 5. 문제 ③ — 내 예측 공식이 틀렸다

프리웜을 붙이고 세 arm 을 같은 지점(비율 1, s=1)에서 돌렸다.

| arm | 히트율 | 캐시 교집합 | 노드 부하 분포 |
|-----|--------|------------|--------------|
| **A** | **1.0000** | — | [1.000] |
| **B** | **0.8996** | **0.622** | [0.500, 0.500] |
| **C** | **1.0000** | **0.000** | [0.525, 0.475] |

**스모킹 건이 잡혔다.** C 의 교집합이 정확히 0 — 키 공간이 완벽히 분할됐고 히트율이
A 와 같은 100% 로 복구된다. B 는 교집합 0.622 로 중복이 실재하고 손실이 10.04%p 다.

**그런데 내 예측은 7.5%p 였다.**

같은 시각에 문헌 조사가 돌아왔고, 정확히 이걸 설명했다:

> `히트율 ≈ ln(C)/ln(N)` 은 **LRU 가 아니라 "상위 C 개를 항상 담고 있는 static
> oracle 캐시"** 의 히트율이다. 실제 LRU 는 미래를 모르니 뜨거운 키도 한동안 안
> 불리면 축출한다 — 그래서 항상 이보다 나쁘다. Berthet(2017)은 α=1 에서 LRU 의
> miss rate 가 static 의 **최대 1.43배**임을 보였고, Jelenković(1999)는 다른
> 체제에서 **e^γ ≈ 1.78배** 상한을 증명했다.

문헌이 쓰라는 건 **Che's approximation** 이었다. 구현해서 대조했다:

| | 실측 | **Che 근사** | 오차 | static envelope (내 원래 예측) |
|---|------|------------|------|------------------------------|
| A | 1.0000 | 1.0000 | 0.00%p | 1.0000 |
| **B** | **0.8996** | **0.8986** | **0.10%p** | 0.9292 (**+2.96%p 낙관**) |
| C | 1.0000 | 1.0000 | 0.00%p | 1.0000 |
| **손실 A→B** | **10.04%p** | **10.14%p** | **0.10%p** | 7.08%p |

**0.1%p 이내로 맞았다.**

이 표가 세 가지를 동시에 말한다. 첫째, **리그가 제대로 돈다** — 이론이 0.1%p 이내로
맞는다는 건 LRU 구현·Zipf 생성기·카운터·라우팅이 전부 의도대로 동작한다는 강력한
간접 증거다. 둘째, **Che 근사가 우리 조건에서 유효하다** — 이제 매 지점의 예측값을
계산할 수 있고, 이게 그동안 비어 있던 **기준소스**다. 셋째, **내 근사가 낙관 편향이라는
문헌의 지적이 실측으로 확인됐다.**

세 번째가 특히 값지다. 문헌 조사와 실측이 **독립적으로** 같은 결론에 도달했다.
문헌만 봤으면 "그런가 보다" 였고, 실측만 봤으면 "예측이 왜 빗나갔지?" 였을 것이다.
둘이 만나서야 "내 근사가 틀렸고, 이만큼 틀렸고, 이유는 이것" 이 됐다.

그리고 이게 사전 등록이 왜 중요한지도 보여준다. 만약 실측을 먼저 보고 나서 예측을
썼다면 나는 십중팔구 Che 를 찾아 쓰고 "예측대로다"라고 적었을 것이다. **틀렸던
7.08% 는 기록에 남지 않았을 것이고, 그러면 "static envelope 을 쓰면 안 된다"는
교훈도 사라진다.**

## 6. 이론이 실험 설계를 바꿨다

Che 근사가 검증되자 전 지점의 예측을 계산할 수 있게 됐다. 그러자 **런을 돌리기
전에** 두 가지가 보였다.

### s=2 는 죽은 구간이다

| 비율 | A | B | 손실 |
|-----|---|---|------|
| 전 구간 | 0.9999~1.0 | 0.9998~1.0 | **0.01%p** |

쏠림이 너무 세서 뜨거운 몇 개가 작은 캐시에도 전부 들어간다. 캐시를 쪼개든 말든
차이가 없다. **"중복 = 기능" 명제의 극단적 확인이긴 한데, 15 런을 돌려서 확인할
가치는 없다.** 계산으로 나오는 걸 측정하지 않는다는 원칙이 그대로 적용된다.

함정이 실제로 최대가 되는 곳은 **s ≈ 0.5** 였다(비율 4 에서 15.42%p) — 균등(s=0)의
12.50%p 보다 **크다**. 반직관적인데, 약한 쏠림이 있으면 A 의 큰 캐시가 그 쏠림을
활용해 히트율을 올리는 반면 B 의 작은 캐시는 같은 쏠림을 덜 활용하기 때문이다.
완전 균등에선 양쪽 다 활용할 쏠림이 없어서 차이가 순수하게 용량 비율로만 난다.

### B 와 C 의 교차점은 히트율에 나타나지 않는다

이게 더 중요한 발견이다.

| s | 함정 크기 (A→B 손실) | arm C 최대 노드 부하 | 1등 키가 먹는 몫 |
|---|---------------------|---------------------|----------------|
| 0 | 12.50%p | 50.2% | 0.00% |
| **0.5** | **15.42%p (최대)** | 50.4% | 0.25% |
| 1.0 | 8.25%p | 52.5% | 8.95% |
| 1.5 | 0.47%p | **69.6%** | 38.4% |
| 2.0 | 0.01%p | **83.5%** | 60.8% |
| 3.0 | — | **95.2%** | 83.2% |

두 열이 정반대로 움직인다. 함정 크기는 s 가 커지면 사라지고, C 의 부하 쏠림은
s 가 커지면 폭발한다. s=2 면 1등 키 하나가 트래픽의 60% 를 먹고, 키 해시는 그 키를
한 노드에 고정하므로 그 노드가 84% 를 혼자 받는다. 라운드로빈이면 정확히 반씩
나눠 받았을 부하다.

**그래서 s ≈ 1.2~1.5 어딘가에서 뒤집힌다.** 아래쪽은 C 가 이기고, 위쪽은 B 가 이긴다.

**그런데 이 교차가 히트율 지도에는 안 보인다.** 고 skew 에서 B 와 C 는 둘 다 100%
라 히트율로는 구분이 안 되기 때문이다. C 의 대가는 **처리량과 지연**으로 나온다 —
핫 노드가 먼저 포화하니까. 즉 교차점은 **Phase 4(부하 구간)에서만 관측된다.**

원래 나는 이 둘을 같은 축에 놓고 "skew 를 올리면 B/C 가 뒤집힌다"고 말해왔는데,
이론이 그게 히트율 축에서는 불가능하다는 걸 알려줬다. 페이즈 설계를 고쳐야 한다.

**부수적으로 부하 쏠림 예측이 실측과 맞았다** — s=1 에서 예측 [0.475, 0.525],
실측 [0.525, 0.475]. 순서가 다른 건 nginx 의 실제 해시함수 대신 md5 로 근사해서
어느 노드가 핫키를 받는지가 달라서고, **크기는 정확히 일치**한다.

## 7. 사전 등록에 생긴 구멍 (정직하게)

원래 규칙은 페이즈의 `spec.md`(예측)를 **커밋한 뒤** 런에 들어가는 것이었다.
커밋 타임스탬프가 "이 예측은 결과를 보기 전에 쓰였다"의 유일한 증거이기 때문이다.

**Phase 1 은 그 규칙을 지키지 못했다.** 훅이 docs 단독 커밋을 막았고(이번 턴에
사용자의 명시 승인이 없음) 사용자는 자리에 없었다. 그래서 `spec.md` 는 커밋 없이
작성됐고, 순서의 증거는 대화 트랜스크립트뿐이다. 이건 원래 규칙보다 **약한 증거**다.

"내가 결과를 안 보고 썼다"는 증거가 아니다. 그게 증거였으면 애초에 커밋 규칙을
만들 이유가 없었다. **Phase 1 의 사전 등록은 "약한 증거"로 표시하고**, 아침에
승인받아 커밋한 뒤 Phase 2 부터는 규칙대로 커밋 → 실행 순서를 지킨다.

## 10. 문제 ④ — 워밍업 k6 가 측정 중에도 계속 돌았다

교차 검증(앱 miss 합 ≡ origin 요청 수)이 **±0.04%** 어긋났다. 카운터 버그라면 한
방향으로 쌓여야 하는데 부호가 갈렸다(+76 부터 −18 까지). 그래서 타이밍 문제로 의심했다.

첫 가설은 **미스를 세는 시점**이었다. 앱은 `cache.get()` 에서 미스를 세고 **그 다음에**
origin 을 호출하므로, 그 사이의 요청은 "앱은 미스로 셌지만 origin 은 아직 못 본" 상태다.
부호가 갈리는 걸 설명한다. 코드로 확인했고 맞았다 — **그런데 크기가 안 맞았다.**
500 RPS 에 미스율 50% 면 in-flight 가 1~2 개인데 ±76 은 너무 크다.

그래서 두 번째를 확인했고, 이게 진짜였다.

| 시점 | APC 의 실행중 k6 컨테이너 |
|------|------------------------|
| `proc.terminate()` 전 | 1 |
| **`proc.terminate()` 후 (5초 대기)** | **1** |

**`Popen(ssh ...).terminate()` 는 로컬 ssh 클라이언트만 죽인다.** 원격의 `docker run` 은
SSH 연결이 끊겨도 계속 돈다 — TTY 를 안 잡았으니 SIGHUP 이 전파되지 않는다.

즉 plateau 에 도달해서 워밍업을 끝냈다고 생각한 순간에도 **워밍업 k6(3,000 RPS)가
계속 돌고 있었고, 측정 60초 내내 측정 k6(500 RPS)와 나란히 BPC 를 때렸다.**
실제 부하는 500 이 아니라 **~3,500** 이었다.

**히트율은 무사하다.** 두 스트림이 같은 키 공간에서 같은 분포로 뽑기 때문이다.
그리고 이건 추측이 아니라 **데이터가 증명한다** — 전 15 지점이 이론과 0.28%p 이내로
맞았고, 부하 오염이 히트율을 흔들었다면 이렇게 맞을 수가 없다. 공교롭게도 이 결함이
"히트율은 RPS 에 무관하다"(`DECISIONS.md` §15)는 주장을 **의도치 않게 실증했다** —
500 과 3,500 이 섞였는데도 이론값이 나왔다.

**하지만 Phase 4 였다면 통째로 망했다.** 거기선 RPS 가 독립변수다. 80/90/100/110% 를
재려는데 정체 모를 3,000 RPS 가 얹혀 있었으면 모든 숫자가 거짓이 됐을 것이다.
**그리고 조용히 망했을 것이다** — 숫자는 여전히 그럴듯하게 나왔을 테니까.

수정: 컨테이너에 이름을 주고 `docker kill` 로 직접 죽인 뒤 **정말 죽었는지 확인하고
아니면 예외를 던진다.** 커밋 `633d7fb`.

## 11. 문제 ⑤ — 사전 등록한 예측이 또 틀렸다 (그리고 이번 게 더 아프다)

`spec.md` 에 이렇게 못 박아뒀었다:

> 균등 분포에선 뜨거운 키가 없으므로 두 캐시는 상당히 다른 키를 담을 것이다.
> **예측: B 의 교집합 비율 ≈ 0.1~0.3.** 0.6+ 가 나오면 내 이해가 틀린 것이다.

실측은 비율 1 에서 **0.4916**, 비율 0.5 에서 **1.0000** 이었다. 틀렸다.

내가 놓친 건 **우연한 겹침**이다. 두 노드가 각각 N 개 중 5,000 개를 무작위로 담으면
서로 다른 걸 담으려 "노력"해도 기대 교집합이 `5000 × 5000/N` 이다. 비율 1 이면 0.5.
**내가 예측한 0.1~0.3 은 이 기저선보다 낮은, 애초에 불가능한 수였다.**

기저선과 나란히 놓으면 전부 붙는다:

| 비율 | B 실측 교집합 | 무작위 기저선 (5000/N) | 초과분 |
|-----|-------------|----------------------|--------|
| 0.5 | 1.0000 | 1.0000 | +0.00%p |
| 1.0 | 0.4916 | 0.5000 | −0.84%p |
| 2.0 | 0.2572 | 0.2500 | +0.72%p |
| 4.0 | 0.1274 | 0.1250 | +0.24%p |
| 8.0 | 0.0674 | 0.0625 | +0.49%p |

**균등 분포에서 B 의 교집합은 100% 우연이다. 핫셋 중복이 아예 없다.**

이게 단순히 "예측 하나 빗나갔다"가 아닌 이유는, **분석 도구가 잘못돼 있었다는 뜻**이기
때문이다. 나는 교집합 수치를 이 프로젝트의 **스모킹 건**이라고 불러왔고
`ARCHITECTURE.md` §5 에 "B 에서 교집합 ≈ 100% 가 나와야 메커니즘의 증거"라고 적었다.

**그런데 교집합 수치는 우연과 중복을 구분하지 못한다.** 비율 0.5 에서 B 의 교집합은
100% 인데 손실은 0%p 다. 교집합이 100% 인데 함정이 없다 — 두 노드가 전 키 공간을
각자 다 담았을 뿐이고(축출이 없으니), 그건 낭비가 아니라 여유다.

**의미 있는 신호는 교집합이 아니라 기저선 초과분이다.** 이 렌즈로 s=1 예비 측정을
다시 보면 교집합 0.622, 기저선 0.5, **초과분 +0.122** — 이게 진짜 핫셋 중복이다.
균등에선 0, Zipf 에선 양수. **뜨거운 키가 있어야 중복이 생긴다**가 이제 숫자로 분리된다.

그리고 이건 명제 서술도 고치게 한다. `README.md` 는 "둘이 보는 트래픽 분포가 같으니
둘 다 똑같은 키를 캐싱한다"고 설명하는데, **그건 Zipf 에서만 맞다.** 균등에서 B 가
50%p 를 잃는 건 중복 때문이 아니라 각 노드가 키 공간의 절반밖에 못 담는데 요청은
아무 노드에나 떨어지기 때문이다. **메커니즘이 다르다.**

**이게 사전 등록의 값어치다.** 결과를 먼저 봤다면 "교집합 0.49 — 역시 중복이 상당하군"
하고 넘어갔을 것이다. 0.1~0.3 이라고 못 박아뒀기 때문에 빗나간 게 눈에 띄었고,
왜 빗나갔는지 파다가 **분석 렌즈 자체가 틀렸다는 걸** 알았다.

## 12. 문제 ⑥ — 부하 생성기를 죽인 건 CDF 만이 아니었다 (APC 호스트 다운)

2026-07-18 밤, CDF 결함(문제 ④ 커밋 f8d037f)을 고친 probe.js 를 재실행했다.
이번엔 VU 초기화가 1,730 근처에서 정체하더니 SSH 가 broken pipe 로 끊겼고,
곧 **APC 가 ping 조차 안 받는 상태**가 됐다. k6 프로세스가 죽은 게 아니라
**호스트가 얼었다.** 다음 날 전원 재시작으로만 복구됐다.

원인은 probe.js 의 시나리오 설계다:

```js
STEPS.forEach((rps, i) => {
  scenarios[`s${rps}`] = {
    executor: 'constant-arrival-rate',
    ...
    preAllocatedVUs: Math.min(3000, Math.max(100, rps)),
    maxVUs: 6000,
```

계단 5개(5k~25k)가 **각자** 시나리오라서 preAllocatedVUs 가 3,000 × 5 = **15,000 VU 를
초기화 단계에서 한꺼번에** 만든다. CDF 는 SharedArray 로 공유되지만 **VU 런타임
자체의 메모리**(VU 당 수 MB)가 곱해져 15G 램을 넘겼다. 게다가 APC 에는 다른 상주
서비스(web·readlog·rag — 약 6G)가 있어 여유는 처음부터 9G 뿐이었다.

**문제 ④와 ⑥은 같은 증상(부하 생성기 사망), 다른 결함이다.** ④는 VU 당 데이터
중복(CDF), ⑥은 VU 개수 자체. ④를 고치고 "이제 된다"고 믿은 게 함정이었다 —
수정 후 재실행이 **더 깊이** 죽었다(프로세스 사망 → 호스트 사망).

교훈 두 개:

1. **부하 생성기도 SUT 다.** 측정 대상만 자원 계획을 하고 생성기는 "알아서 되겠지"
   했다. 생성기의 램·CPU 도 측정 설계의 입력이고, 안전장치(`docker run --memory` 캡)
   없이 돌리면 최악의 실패 모드는 "느려짐"이 아니라 **호스트 통째 소실**이다.
2. **arrival-rate 실행기의 preAllocatedVUs 는 지연 가정의 함수다.** 25k RPS 라도
   지연 1ms 면 VU 는 ~25개면 된다. 3,000 은 "포화로 지연이 120ms 까지 튀어도 버틴다"
   는 방어인데, 그 방어를 계단 수만큼 곱해 두는 순간 방어가 흉기가 된다.

수정 방향은 probe.js 를 고치는 대신 **계단을 k6 런 분리로 구현**하는 것으로 바꿨다
(`load.js` 를 RPS 만 바꿔 순차 실행). 런마다 VU 풀이 해제되고(preAlloc ≤ 2,000),
컨테이너 메모리 캡으로 최악에도 호스트가 아니라 컨테이너만 죽는다.
사전 등록: `docs/phases/phase-0-saturation/spec.md` §3.1.

**부수 발견 — 조종 노트북은 부하 생성기가 될 수 없다.** APC 가 죽어 있는 동안
노트북(24C/36G)을 대체 생성기로 검증했는데 실격이었다: 8,000 RPS 목표에 4,623 달성,
드롭 42%. 원인은 성능이 아니라 **경로**다 — 노트북→BPC 는 WiFi(wlo1) + 라우터 홉이라
RTT 3ms±1.6, APC 의 유선 sub-ms 와 자릿수가 다르다. 코어가 아무리 많아도
네트워크 경로가 측정 장비 스펙이다.

## 14. Phase 0·4 — 무릎과 역전 (s=1.5 실측)

전날의 질문 — "B·C 교차점은 히트율이 아니라 포화점에서 보인다" — 이 실측으로 닫혔다.
상세는 `docs/phases/phase-0-saturation/`·`phase-4-load/` 의 spec → result → gate.

**Phase 0 (무릎)**: 생성기 천장(10.7k)을 실측하고 BPC CPU 를 총량 0.5 로 조여
무릎을 그 아래로 내렸다. 세 arm 의 무릎 = A 5000 / B 4500 / C 4000. 백미는 산수다:
0.25CPU 노드 하나의 용량 ≈ 2,500 rps 인데, **C 의 무릎에서 핫 노드가 받는 부하 =
4000 × 0.628 = 2,512.** 키 해시의 쏠림은 정확히 그만큼의 용량 손실로 번역된다.

**Phase 4 (80~120% 지도)**: 히트율 축에선 A−B ≤ 0.4%p — s=0 에서 50%p 였던 중복의
세금이 사실상 소멸. 대신 C 가 5000 부터 p50 16ms(핫 노드 스로틀 큐), 6000 에선 95ms.
같은 지점 B 는 0.38ms·43ms. **s=0 의 정답이 s=1.5 의 오답이 된다** — 같은 배치 결정이
접근 분포에 따라 반대로 채점되고, A(스케일업)는 양쪽 함정 모두에 면역이다(단일
장애점이라는 별도 청구서를 받을 뿐).

사전 등록 예측 8개의 성적: 온전한 일치는 3개뿐이다(phase-0 P3 · phase-4 P3·P4).
P1 은 두 페이즈 모두 빗나갔고, phase-0 P2 는 **런 선택에 민감**(등록 프로토콜 20s 런은
빗나감, 30s 재측정 런만 일치 — gate 에 병기), phase-4 P2·phase-0 P4 는 부분. 전부
원인이 특정됐고 위 표의 #10·#11 과 "워밍업 시드 ≠ 측정 시드"(phase-4 result 특기 3)로
환원된다. 그리고 판정 문서 자체도 듀얼 리뷰(독립 Claude 워커 ∥ codex)에서 8건을
지적받아 고쳐졌다 — 지표 사후 교체·인용 숫자 오류까지 포함해서. 빗나간 예측이
이번에도 맞은 예측보다 많은 것을 가르쳤다.

## 15. arm D — 캐시를 밖으로 빼봤다 (외부화, Phase D)

같은 날 오후, TODO 3번을 끝까지 밀었다. 2 노드 라운드로빈 + **공유 Redis 캐시 하나.**
Redis 의 기본 축출(maxmemory allkeys-lru)은 바이트 기준 + 샘플링 근사라 쓰지 않고,
HASH+ZSET+단조 카운터를 Lua 로 묶어 **LruCache 와 같은 엔트리 정확 LRU** 를 구현했다
(같은 이유로 Caffeine 을 버렸던 결정의 연장). 용량 3 시나리오로 축출 순서까지 검증한
뒤에야 측정에 들어갔고, 이번엔 선행 페이즈의 절차 교훈 3개(30s 명시·복합 판정 규칙
사전 등록·측정 시드 워밍업)를 spec 에 먼저 박았다.

결과는 예측 4개 중 3개 일치(P1 히트율 ≡ A · P3 무릎 구간+병목=앱 CPU · P4 분포),
1개는 좋은 쪽으로 빗나감(P2 — 지연 세금 +0.08ms, 예측 하한 0.5ms 아래). 종합:

**외부화의 청구서는 지연이 아니라 용량으로 온다.** 무릎 3000 — A(5000)의 60%.
왕복 세금은 median 에 +0.08ms 로만 보이는데, 무릎은 예산 세금(redis 몫 CPU −20%) ×
효율 세금(동기 호출의 앱 측 비용 — 무릎 기준 per-CPU −17%)이 곱해져 깎였다. 덤으로
4 arm 중 유일하게 과부하에서 **goodput 이 역행**하는 험한 붕괴를 보였다. 이게 측정
아티팩트(k6 스텝말 절단)인지 실체인지 4000 RPS × 180s 연속 런으로 판별했다 —
**실체였고, 진행성이었다**(달성 3.0k→1.2k, 실패 67%, admin 경로까지 무응답 =
스레드 전원 고갈 지속). 나이브 외부화는 무릎을 넘기면 시간이 지날수록 무너진다.

이로써 개념 지도가 닫혔다: **같은 자원, 네 가지 배치, 네 가지 함정** — B 히트율 함정
(s=0 에서 50%p) · C 쏠림 함정(고 skew 에서 용량 −15%) · D 용량 함정(−40%) · A 는 숫자
밖의 함정(단일 장애점). 어느 함정이 아픈지는 접근 분포(s)와 부하가 정한다.
