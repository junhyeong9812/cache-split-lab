# Phase 0 — 판정 (gate)

> 입력: spec.md 의 사전 등록 예측 P1~P4 · result.md 실측.

## 방법 이탈 명시 (사전등록 대비)

spec §3.1 은 계단을 **20s** 로 등록했다. 실제 판정 런(run2)은 **30s** 다 — 20s 런(run1)에서
B 5500 비단조 이상치(드롭 2.8% → 6000 정상)를 보고 창만 늘려 재측정했다. **이건 결과를
본 뒤의 방법 변경이고, 사전등록 절차 위반으로 기록한다.** 판정 기준(①②③)은 안 바꿨고,
run1 무릎(A 5000·B 4000·C 4000)도 result.md 에 병기했다 — **P1 비율은 run1 로 계산해도
1.00, run2 로도 0.889, 어느 쪽도 등록 밴드(0.62~0.82) 안이 아니므로 P1 판정(빗나감)은
런 선택과 무관**하다. **그러나 P2 는 런 선택에 민감하다** — 아래 P2 절에 병기.
이후 페이즈는 계단 길이를 spec 에 30s 로 등록하고 시작한다.

## P1 — knee_C / knee_B ≈ 0.72 (밴드 0.62~0.82): **빗나감**

실측: 4000 / 4500 = **0.889** — 밴드 밖.

**왜 빗나갔나 — 예측의 분할 가정이 틀렸다.** 0.72 는 "패리티(홀짝) 분할이면 핫 노드가
부하 0.692 를 진다"에서 유도했는데, **실제 nginx 해시 분할의 핫 노드 부하는 0.628** 이었다
(전 계단에서 0.628~0.630 으로 놀랍도록 안정 — raw 의 노드분포 열). top-1 키(질량 0.384)와
top-2 키(질량 0.136)가 **다른 노드에 떨어졌다**: 0.384 + 나머지의 랜덤 몫 ≈ 0.63.
실측 쏠림으로 재계산하면 0.5/0.628 = **0.796** — 그리드 해상도(500 단위 계단이 비율에
±0.05~0.1 양자화를 만든다) 안에서 실측 0.889 와 붙는다.

**메커니즘 자체는 정량적으로 닫혔다** — 이게 이 페이즈의 진짜 수확:

| | 유도 | 값 |
|---|------|-----|
| 0.25CPU 노드 1개 용량 | A 무릎 5000 의 절반 (A 는 0.5CPU) | ~2,500 rps |
| C 무릎에서 핫 노드 부하 | 4000 × 0.628 | **2,512 rps** |
| B 무릎에서 노드당 부하 | 4500 × 0.5 | 2,250 rps |

**닫힌 것과 열린 것을 구분한다.** 닫힌 것: **C 의 산수** — A 에서 유도한 노드 용량(2,500)
대비 C 핫 노드의 무릎 부하(2,512)가 0.5% 안에서 맞는다("쏠림 = 용량 손실"의 환율).
열린 것: **B·C 노드 용량 동일 가정** — B 는 노드당 2,250 에서 꺾여 C 노드(2,512)와 12%
다르다. 즉 P1 유도의 가정 하나(R_node 동일)는 이 데이터로 입증되지 않았다. B 의 추가
미스(히트 0.996 vs 0.9995 — 미스는 5ms origin 왕복 동안 스레드를 점유)가 후보 원인이나
이 페이즈 데이터로는 단정하지 않는다.

**교훈 (틀린 것 #10 후보)**: "해시 분할의 쏠림"은 이론 상수가 아니라 **해시 함수와 키
이름의 함수**다. 예측하려면 분할을 계산하지 말고 실측하라 — 부하 분포는 계단 전에
저부하로 잴 수 있었다.

## P2 — knee_B ≈ knee_A ±15%: **조건부 일치 (런 선택에 민감 — 정직하게 병기)**

판정 런(run2, 30s): 4500/5000 = 0.90 (−10%) → 밴드 안. **그러나 사전 등록 프로토콜
그대로의 런(run1, 20s)에선 4000/5000 = 0.80 (−20%) → 밴드 밖 = 빗나감.** 30s 재측정
결정은 결과(B 이상치)를 본 뒤였으므로, run2 만 인용해 "일치"로 적으면 사후 선택이
판정을 fail→pass 로 뒤집은 사실이 숨는다. 판정 기준 자체가 아니라 런 선택이 결론을
가르는 상태 — **"일치"는 30s 런 한정이며, 확정하려면 30s 프로토콜 사전 등록 반복이
필요**하다. 무릎과 별개로 최대 달성 처리량은 두 런 모두 B ≈ A (6,891 vs 6,982, −1.3%) —
"스케일아웃 자체의 세금 ≈ 0"의 방향은 런 무관하게 지지된다.

## P3 — 무릎의 병목 = 앱 CPU: **일치**

무릎 시점 앱 컨테이너가 쿼터 소진(A 53%/0.5 · B 26·27%/0.25 · C 핫 25%/0.25),
origin ≤8%/400%, nginx ≤43%(비제한), APC ≤70%(무릎 구간 ~50%). 빚 B(무엇이 묶는가) 해소:
**CPU 쿼터**가 묶고, 스로틀링이 지연 폭발(p50 0.3ms → 수십 ms)로 나타난다.

## P4 — 무릎 전 세 arm p95 ±50%: **부분 빗나감**

p50 은 동등(0.356/0.372/0.364 — ±5%). p95 는 A 0.80 / B 5.96 / C 0.99 — B 만 7배 이탈.
원인: B 는 같은 RPS 에서 미스가 ~10배 많고(히트 0.989 vs 0.999) 미스는 5ms origin 왕복이라
**p95 가 미스 꼬리를 그대로 노출**한다. 즉 이건 "노이즈"가 아니라 **B 의 히트율 손실이
지연 꼬리로 번역된 것** — Phase 1 미해결 ②(B 가 C 보다 60% 느림)와 같은 메커니즘으로 보인다.
예측이 "p95 동등"이라 쓴 게 히트율 손실의 지연 표현을 잊은 것.

## 종합 — 2번 질문(B·C 교차점)에 대한 답

**s=1.5 에서 역전은 실재한다.** 히트율 축에선 C 승(0.9995 vs 0.996 — 그러나 차이 0.35%p 로
사실상 소멸), **용량 축에선 B 승** (무릎 4500 vs 4000, 최대 처리량 6,891 vs 5,889 — C 가
-15%). s=0 에서 "라운드로빈 중복 = 50%p 손실"이던 것이, 고 skew 에선 "키 해시 쏠림 =
용량 -15%"로 부호가 뒤집힌다. **"라운드로빈이 항상 진다"는 s=0 의 일반화 오류였다**
(README 미해결 ③ 해소).

## Phase 4 입력

- **100% 기준선 = A 무릎 5000** (자 arm 기준).
- 그리드: 80/90/100/110/120% = 4000/4500/5000/5500/6000 — 120% 를 포함해야
  "B 는 버티고 C 는 무너지는" 구간(6000)이 들어온다.
- 각 지점 60s, 세 arm 동일 절차. 예측은 phase-4 spec.md 에 사전 등록.
