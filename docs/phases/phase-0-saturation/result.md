## 환경 (spec §3.2·§3.3 캘리브레이션 결과)

| 항목 | 값 | 근거 |
|------|----|------|
| 부하 생성기 | **APC** (192.168.55.9, 8T/15G) — 전 구간 통일 | 노트북은 실격(WiFi 경로 RTT 3ms±1.6, 8k 목표에 4.6k 달성 — spec §3.3) |
| APC 천장 | **~10.7k RPS** (k6 컨테이너 CPU 775%/800% 포화) | arm A CPU=2 계단(5k~25k): 15k 부터 호스트 795% 포화, 달성 10.7k 상한 |
| **CPU 총량 (조임 확정)** | **0.5** — A: 0.5×1 / B·C: 0.25×2 | CPU=2 에선 무릎 ≫ 천장 (앱 78~84%/200% 에서 APC 먼저 포화). 앱 실측 소모 ~7.6%/1k RPS → 0.5 면 무릎 ~6.6k < 천장 70% |
| 불변 유지 | 캐시 10,000 (A: 10,000 / B·C: 5,000×2) · 스레드 200 (A: 200 / B·C: 100×2) · origin 4CPU·5ms · N=40,000 · s=1.5 · SEED=42 | run_arm.py 불변식과 동일 구조, CPU 만 4.0→0.5 |
| k6 안전장치 | 계단 = **런 분리**(probe.js 미사용) + 컨테이너 `-m 6g` 캡 | spec §3.1 (TIMELINE 문제 ⑥) |
| ollama | 전 구간 유휴 (BPC loadavg 정상) | 계단 중 BPC 컨테이너 CPU 샘플링 로그 |

절차(각 arm 동일): `up -d --force-recreate` → 프리웜(40k 전 키 1회) → s=1.5 워밍업(2,000 RPS·60s)
→ 카운터 리셋 → 계단(각 30s, 계단 간 4s·노드별 카운터 스냅샷). 1차 런(20s 계단)은 B 5500 에서
비단조 이상치(드롭 2.8% → 6000 은 정상)가 나와 **계단 길이만 30s 로 늘려 재측정**했다 —
판정 기준(spec §3.1)은 불변. 20s 단일 계단은 CFS 스로틀링 진동에 짧다.
