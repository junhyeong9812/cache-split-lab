## 1. 질문

세 배치(A 단일 / B 라운드로빈 / C 키 해시)는 **각각 몇 RPS 에서 무릎 꿇는가.**

히트율 지도(Phase 1~3)에서 C 는 공짜처럼 보인다 — 히트율 손실 0. 그러나 고 skew 에서
C 는 뜨거운 키를 가진 **한 노드에 부하가 쏠린다.** 그 대가는 히트율 축이 아니라
**용량(포화점) 축**에서 청구된다. Phase 0 은 그 축을 잰다. 이 숫자가 Phase 4
(80/90/100/110% 부하 구간)의 100% 기준선이 된다.

**s=1.5 를 재는 이유**: 이론상 B 의 히트율 손실이 0.47%p 로 거의 소멸하는 반면
C 의 부하 쏠림은 69.2% 로 커지는 — **B·C 우열이 뒤집힐 것으로 예측되는** 지점이다
(N=40,000 기준. `scripts/theory.py` Che + top-1 질량 계산, 2026-07-19).

## 2. 사전 등록된 예측

N=40,000 · s=1.5 · 캐시 총량 10,000 (A: 10,000 / B·C: 5,000×2) · CPU 총량 고정(값은 §3 캘리브레이션 후 기록).

이론 입력값 (`scripts/theory.py`, Che's approximation + Zipf 질량):

| | A | B | C |
|---|---|---|---|
| 히트율 | 0.9942 | 0.9894 | ≈0.9942 |
| 노드 부하 분포 | 1노드 | 0.50/0.50 | **0.692/0.308** (top-1 키 질량 0.384 가 한 노드에) |

**P1 (주 예측 — CPU 총량 무관 비율)**: C 의 포화 RPS ÷ B 의 포화 RPS ≈ **0.72**.
근거: B·C 노드는 CPU 가 같고 히트율도 사실상 같으므로(0.9894 vs 0.9942) 노드 한 개의
처리 한계 R_node 는 동일. B 는 R_total×0.5 가, C 는 R_total×0.692 가 그 한계에 먼저
닿는다 → knee_C/knee_B = 0.5/0.692 = 0.723.
**판정 기준: 0.62 ≤ 비율 ≤ 0.82 면 일치** (±0.10 — 무릎 판정 자체의 오차 감안).

**P2**: B 의 포화 RPS ≈ A 의 포화 RPS (**±15%**). 근거: A 는 노드 1개에 CPU 2배 —
per-핵 처리량이 같다면 총 한계는 같다. 미스율 차이(0.6% vs 1.1%)는 origin 여유(4CPU,
5ms) 안에서 흡수된다. 이게 깨지면(예: B ≪ A) 스레드 풀·컨텍스트 스위칭 등
CPU 외 병목이 있다는 뜻 — 그 자체가 발견이다.

**P3**: 무릎에서 묶이는 자원은 **앱 CPU** (nginx·origin·APC 아님). 판정: 무릎 시점
BPC 앱 컨테이너 CPU ≥ 배정량의 90%, origin CPU 는 배정량의 50% 미만, 부하 생성기
CPU 70% 미만.

**P4 (음성 대조)**: 무릎 이전 구간(≤80%)에서 세 arm 의 p95 는 서로 ±50% 이내 —
포화 전에는 배치 차이가 지연에 거의 안 보인다.

## 3. 방법

### 3.1 측정 규약

- 무릎 정의(사전 고정): RPS 계단(각 20s)을 올리며 **① dropped_iterations > 1% 또는
  ② p95 가 최저 계단 대비 10배 초과 또는 ③ http_req_failed ≥ 0.1%** 가 처음 발생하는
  계단. 무릎 RPS = 그 직전 계단.
- 계단은 **k6 런 분리로 구현한다** (`load.js` 를 RPS 만 바꿔 순차 실행, 각 런 사이 5s).
  `probe.js`(멀티 시나리오 1런)는 **쓰지 않는다** — 시나리오당 preAllocatedVUs 3,000
  × 5단계 = 최대 15,000 VU 가 초기화 단계에서 부하 생성기 램(15G)을 넘겨 **호스트가
  얼었다** (2026-07-18 실측, exit 255 · no route to host). 런 분리는 VU 풀을 계단마다
  해제하므로 동일 측정을 안전하게 얻는다.
- k6 컨테이너는 `--memory` 캡을 걸어 최악에도 컨테이너만 죽게 한다(호스트 보호).
- 각 계단 전 프리웜 + s=1.5 워밍업으로 정상상태 진입(러너 관례와 동일). 계단 간에는
  캐시를 유지한다(정상상태 연속).
- 런마다 BPC·부하 생성기 CPU 를 5s 간격 샘플링(P3 판정 입력).

### 3.2 캘리브레이션 (측정 아님 — 예측 판정에 안 씀)

부하 생성기 천장과 BPC 무릎의 위치를 맞추는 튜닝 구간:
1. 부하 생성기가 안정적으로 뽑는 최대 RPS 확인 (BPC 무관하게 결정되는 값).
2. arm A 무릎이 그 천장의 70% 아래로 오도록 **CPU 총량을 조여서 고정** (예: 4.0 → 1.0).
   조인 값은 세 arm 에 동일 적용(A: x / B·C: x/2)하고 result.md 에 기록한다.
3. 어느 조합으로도 무릎 < 천장을 못 만들면 "이 하드웨어로 고 skew 포화 측정 불가"를
   근거와 함께 기록하고 Phase 4 를 재설계한다.

### 3.3 환경 각주 (사전 기록)

- 2026-07-18 프로브 사고로 APC(192.168.55.9)가 다운됐다. 재부팅 전이면 부하 생성기를
  **조종 노트북(192.168.55.114, 24C/36G)** 으로 대체한다. Phase 0/4 안에서는 어느
  쪽이든 **한 대로 통일**한다(중간 교체 금지). 사용한 생성기를 result.md 에 기록.
- BPC ollama(11434)는 유휴 확인 후 진행, 런마다 loadavg 감시.
