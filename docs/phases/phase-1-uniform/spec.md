## 1. 질문

캐시를 가진 티어를 복제하면 히트율이 얼마나 무너지는가 — **접근이 완전히 균등할 때.**

균등 분포는 캐시에 가장 불리한 조건이다. 뜨거운 키가 없으므로 캐시가 "무엇을
남길지" 고를 근거가 없고, 히트율은 순수하게 **용량 대비 키 공간의 비율**로만 결정된다.
그래서 여기가 함정이 가장 순수하게 드러나는 지점이고, 동시에 **이론이 근사가 아니라
정확한 값을 주는 유일한 지점**이다(문헌: Che 근사는 균등 분포에서 exact).

즉 이 페이즈는 두 가지를 동시에 한다. 함정의 크기를 재는 것과, **리그가 정확한
이론값을 재현하는지 확인하는 것.** 두 번째가 사실 더 중요하다 — 여기서 어긋나면
이후 모든 페이즈의 숫자를 믿을 수 없다.

## 2. 사전 등록된 예측

`scripts/theory.py` — Che's approximation. 균등 분포에서는 `히트율 = min(1, C/N)` 이
정확하고, Che 가 이를 재현하는 것도 확인했다.

캐시 총량 10,000 고정 (A: 10,000 / B·C: 5,000 × 2노드).

| 비율 | 키공간 N | **A 예측** | **B 예측** | **C 예측** | **손실 A→B 예측** |
|-----|---------|-----------|-----------|-----------|-----------------|
| 0.5 | 5,000 | 1.0000 | 1.0000 | 1.0000 | **0.00%p** ← 음성 대조군 |
| 1.0 | 10,000 | 1.0000 | 0.5000 | 1.0000 | **50.00%p** ← 최대 |
| 2.0 | 20,000 | 0.5000 | 0.2500 | 0.5000 | 25.00%p |
| 4.0 | 40,000 | 0.2500 | 0.1250 | 0.2500 | 12.50%p |
| 8.0 | 80,000 | 0.1250 | 0.0625 | 0.1250 | 6.25%p |

### 부수 예측

**노드별 부하 분포:** 세 arm 모두 50/50 (±2%). 균등 분포에는 뜨거운 키가 없으므로
키 해시(C)도 부하를 고르게 나눈다. **C 의 부하 쏠림 비용은 이 페이즈에서 0 이다** —
그건 고 skew 의 이야기다.

**캐시 교집합 (B):** 여기서 예측이 갈린다. 나는 지금까지 "B 는 두 노드가 같은 키를
중복 캐싱한다"고 말해왔는데, **균등 분포에선 그게 성립하지 않을 것 같다.** 뜨거운
키가 없으므로 각 노드의 LRU 는 "자기가 최근에 본 것"을 남기는데, 라운드로빈이면
두 노드가 본 최근 요청이 서로 다르다. 그래서 **두 캐시는 상당히 다른 키를 담을 것이다.**

즉 균등에서 B 의 손실은 **중복 때문이 아니다.** 각 노드가 키 공간의 절반밖에
못 담는데 요청은 아무 노드에나 떨어지기 때문이다. 메커니즘이 다르다.

- **예측: B 의 교집합 비율 ≈ 0.1~0.3** (낮음). 만약 0.6+ 가 나오면 내 이해가 틀린 것이다.
- **예측: C 의 교집합 비율 = 0.000** (완전 분할 — 결정적 라우팅의 직접 증거).

**arm C:** 전 지점에서 A 와 동일. C 는 "캐시 C/2, 키 공간 N/2" 라 비율이 A 와 같다.

## 3. 판정 기준 (결과를 보기 전에 정한다)

| 항목 | 통과 조건 |
|------|----------|
| **이론 일치** | 각 지점의 실측 히트율이 예측 ±2%p 이내 |
| **음성 대조군** | 비율 0.5 에서 A·B·C 모두 ≥ 0.98, 손실 ≤ 2%p |
| **교차 검증** | 앱 miss 합 ≡ origin 요청 수 (정확히 일치) |
| **캐시 정확성** | 각 노드 `cache_size ≤ capacity`, 가득 찬 뒤 `== capacity` |
| **C 분할** | C 의 교집합 = 0 (정확히) |
| **부하 균등** | 세 arm 모두 노드 분포 50/50 ± 2% |
| **에러율** | 0 (아니면 그 지점 지연 지표 무효) |
| **워밍업** | plateau 도달. 미도달이면 그 지점은 "미수렴"으로 표시하고 해석에서 제외 |

**하나라도 깨지면 그 런의 숫자는 버린다.** 특히 음성 대조군이 깨지면 리그 전체가
의심스러우므로 나머지 결과를 보지 않는다.

## 4. 측정 설정

| 항목 | 값 | 근거 |
|------|-----|------|
| 캐시 총량 | 10,000 (A: 10,000 / B·C: 5,000×2) | `DECISIONS.md` §13 |
| CPU 총량 | 4.0 (A: 4.0 / B·C: 2.0×2) | 총 자원 고정 |
| 스레드 총량 | 200 (A: 200 / B·C: 100×2) | `DECISIONS.md` §17 |
| 미스 페널티 | 5ms 고정 | 파라미터지 측정 대상 아님 |
| RPS | 500 | **잠정** — Phase 0(포화점) 미실시. 아래 참조 |
| 측정 길이 | 60s | |
| 프리웜 | 전 키 1회 훑기 | 쿠폰수집가 구간 제거 (§5) |
| 워밍업 | 3,000 RPS, plateau 탐지, 최대 300s | 히트율은 RPS 무관 → 빠르게 채운다 |
| seed | 42 | 재현성 |

**RPS 500 의 근거가 약하다.** Phase 0(포화점 측정)을 아직 안 했으므로 이건
"충분히 낮아 보이는 값"이지 측정된 값이 아니다. 히트율은 RPS 에 무관하므로
(`DECISIONS.md` §15) 이 페이즈의 주 지표는 영향받지 않지만, **지연 지표는
이 페이즈에서 해석하지 않는다.** 그건 Phase 4 의 일이다.

## 5. 이 설정에 들어간 교정 — 프리웜

스모크 테스트에서 문제가 하나 잡혔다. 비율 1 에서 arm A 는 **축출을 아예 안 한다**
(용량 10,000 ≥ 키 공간 10,000). 그래서 A 의 미스는 "처음 보는 키"일 때만 발생하고,
히트율이 100% 에 **점근**한다. 그런데 Zipf 꼬리의 차가운 키가 좀처럼 안 뽑혀서
수렴이 극도로 느리다 — 실측으로 500 RPS × 80초(= 40,000 요청)를 돌려도
10,000 키 중 **6,638 개만** 캐시에 들어왔고 히트율은 0.92 에서 여전히 오르는 중이었다.

수렴 전에 측정하면 A 가 과소평가되고, 따라서 손실(A−B)도 과소평가된다.
**우리 가설에 불리한 방향의 편향**이므로 반드시 제거해야 한다.

해법은 **결정적 프리웜** — 모든 키를 정확히 한 번씩 훑고 시작한다(`k6/prewarm.js`).
이건 치팅이 아니다. 우리가 재려는 건 **정상상태** 히트율이고, 프리웜은 거기 빨리
도달할 뿐 도착점을 바꾸지 않는다. 세 arm 에 똑같이 적용되고, 프리웜 후에도 Zipf
워밍업을 돌려 LRU 가 핫셋 순서를 정리하게 한다.

효과는 즉시 확인됐다 — 프리웜 11.7초에 10,000/10,000 적재, 워밍업 40초에
plateau 도달, 히트율 정확히 1.0000.

## 6. 실행

```bash
for n in 5000 10000 20000 40000 80000; do
  for arm in a-single b-roundrobin c-keyhash; do
    python3 scripts/run_arm.py $arm --keyspace $n --skew 0 --rps 500 \
      --duration 60s --out results/phase-1-uniform/$arm-$n.json
  done
done
```

15 런. arm 은 순차 실행(동시에 띄우면 서로가 서로의 노이즈가 된다).

## 7. 알려진 오염원

**BPC 에 ollama 가 2주째 돌고 있다**(포트 11434, healthy). 사용자의 작업이라 끄지
않았다. 유휴 상태면 CPU 를 안 쓰지만 **실험 도중 누가 질의하면 16 코어를 먹어서
측정을 오염시킬 수 있다.** 런마다 BPC 의 CPU 를 같이 기록해서 움직이면 잡아낸다.

히트율은 CPU 경합에 안 흔들리므로 이 페이즈의 주 지표는 안전하다. Phase 4(부하)
에서는 치명적일 수 있다.
