# Phase D — 판정 (gate)

> 입력: spec.md P1~P4 · result.md. 방법 이탈 1건 명시: 1차 계단의 2000·2500 이 과도
> 상태라 **저부하 4계단만 정상상태에서 재계단**했다(pdr2 — 판정 기준·계단 길이 불변,
> 고부하 계단은 1차 그대로). phase-0 의 교훈대로 두 런 모두 result.md 에 병기.

## P1 — 히트율 ≡ A (±0.3%p): **일치**

D 정상상태 0.9996~1.0000 vs A 0.9993~0.9994 — 차이 ≤ 0.07%p. 캐시 총량 하나(10k)의
연역이 실측으로 확인 = **리그 정합성 게이트 통과** (Lua LRU 가 LruCache 와 같은 일을
하고 있다는 간접 증거이기도 하다 — 직접 증거는 용량 3 축출 시나리오, task.md §3).

## P2 — 저부하 p50 = 0.5~1.5ms: **빗나감 (좋은 쪽으로)**

실측 **0.42~0.43ms** — 등록 구간 아래. **증거가 지지하는 진술은 "종단간 p50 차이가
+0.07~0.09ms(1.2배)였다"까지다** — 분위수는 뺄셈으로 구성요소(RTT·Lua·Lettuce)로
분해되지 않고 두 arm 은 CPU 배치도 다르므로, 그 차이를 "redis 왕복 시간"으로 읽는
것은 비약이다. 확실한 것: 예측이 상정한 0.5~1.5ms 급의 지연 세금은 median 에
나타나지 않았다. **"외부화 = 지연 폭탄"은 이 조건(동일 호스트·브리지)에선 성립하지
않는다** — 단 네트워크 홉이 실물(µs~ms)인 배치로 일반화할 수 없다(랩 한계로 기록).

## P3 — 무릎 3,000~4,000 (점 3,500) + 병목=앱 CPU: **일치**

무릎 **3000** (등록 구간의 하한 — 점 추정 3,500 보다 낮았다).

병목 판정은 등록 문구("**무릎 시점**")대로 **판정 런(pdr2)의 3000 계단 창**에서, 앱과
redis 에 **같은 통계량(max)** 을 적용해 낸다 (1차 초안이 전 구간 통계 + 앱 max/redis
p90 비대칭이었던 것을 리뷰가 잡음 — 아래는 재산출):

| 컨테이너 | 쿼터 | 무릎 계단(3000) max | 쿼터 대비 | 등록 조건 | 판정 |
|---------|------|-----|------|------|------|
| app-1 / app-2 | 0.2 | 18.6% / 19.0% | **93% / 95%** | ≥ 90% | ✓ |
| redis | 0.1 | 6.8% | **68%** | < 70% | ✓ |

(첫 위반 계단 3500 에선 앱 19.6~19.8%(98~99%)로 쿼터에 완전히 붙고 redis 는 5.8%(58%)로
오히려 내려간다 — goodput 감소의 반영.) origin ≤1.7%·nginx ≤25%(비제한)·APC 여유.
**앱 CPU 가 묶는다** — redis 단일 스레드가 아니라. result.md 의 전 구간 CPU 표는
보조 자료로 유지(라벨 명시).

## P4 — 분포 0.50/0.50 (±0.01) 전 계단: **부분 일치 (등록 문구 그대로는 빗나감)**

등록은 "**전 계단** ±0.01"인데 1차 계단의 과부하 구간(5000: 0.516/0.484 · 6000:
0.526/0.474)이 최대 0.026 이탈한다 — "정상 구간만 보면 일치"로 좁히는 것은 지표
사후 축소다(phase-4 P2 를 부분 빗나감으로 기록한 자기 선례와 일관되게 여기도 같은
잣대). **부분 일치**: 무릎 이하 전 계단 0.500±0.005 ✓ / 과부하 계단 ✗. 이탈의 해석:
드리프트가 실패·드롭 비대칭과 동반되는 과부하 아티팩트로 보이나, "과부하에서도
분포가 상수"라는 등록 주장 자체는 데이터가 기각했다. C 형 쏠림(특정 노드 상시 편중)이
아니라는 점은 유지 — 정상 구간 0.500 정확.

## 종합 — 4 arm 지도 (같은 복합 규칙·같은 리그, CPU 총량 0.5)

| | A 스케일업 | B RR 복제 | C 키 해시 | **D 외부화** |
|---|---|---|---|---|
| 무릎 RPS | 5000 | 4500 | 4000 | **3000** |
| 최대 달성 | ~6,980 | ~6,890 | ~5,890 | **~3,480** |
| 저부하 p50 | 0.34ms | 0.37ms | 0.36ms | 0.42ms |
| 히트율(정상) | 0.9994 | 0.996~8 | 0.9993 | 0.9996+ |
| 과부하 거동 | 달성 유지 | 달성 유지 | 달성 유지 | **goodput 역행 (3.5k→1.8k)** |

**외부화의 대가는 지연이 아니라 용량이었다.** 히트율은 A 와 동률(캐시 하나의 연역),
종단간 p50 차이도 +0.08ms 뿐. 그러나 무릎이 B 의 67%(3000/4500) — **같은 기준점끼리
비교한 분해**는 이렇다 (초안의 "per-CPU −3%"는 B 는 무릎, D 는 최대 달성이라는 서로
다른 운전점을 섞은 오계산 — 리뷰가 잡음):

| 기준점 (동일 적용) | B 노드 | D 노드 | 앱 per-CPU 격차 |
|---|---|---|---|
| 무릎 ÷ 노드 ÷ CPU | 4500/2/0.25 = 9,000 | 3000/2/0.2 = 7,500 | **−17%** |
| 최대 달성 ÷ 노드 ÷ CPU | 6890/2/0.25 = 13,780 | 3481/2/0.2 = 8,700 | **−37%** |

무릎 기준 총 손실 −33% ≈ **예산 세금 −20%**(redis 몫 CPU) × **효율 세금 −17%**(동기
왕복의 앱 측 비용 — 작지 않다). 그리고 D 는 무릎 대비 최대 달성의 여유가 1.16배뿐
(A/B/C 는 1.4~1.5배) — **포화가 가파르고, 과부하에선 goodput 이 역행**한다(3.5k→1.8k).
즉 "그냥 redis 쓰면 되잖아"의 청구서 세 줄: CPU 예산 분할 + 동기 호출 효율 + 험한
붕괴 모드.

개념 지도가 닫혔다: A=스케일업 / B=나이브 스케일아웃(히트율 함정) / C=분할
스케일아웃(쏠림 함정) / D=외부화(용량 함정). **네 배치 모두 같은 CPU·캐시 엔트리
예산인데 함정만 다르다** — 단 메모리 총량은 고정하지 않았다(redis 256m 이 앱 mem_limit
축소 없이 추가 — 캐시가 엔트리 수로 강제되므로 메커니즘엔 무영향이나 "같은 자원"의
범위는 CPU·캐시·스레드로 한정해 읽어야 한다). s(접근 분포)와 부하가 어느 함정이
아픈지를 정한다.

## 판별 런 — goodput 역행은 실체다 (2026-07-19 오후, 미해결 ① 해소)

30s 계단의 붕괴 숫자가 스텝말 아티팩트(k6 gracefulStop 5s 가 미완 요청을 절단)일
가능성을 배제하기 위해 **4000 RPS × 180s 연속 단일 런**을 돌렸다
(raw: `results/phase-d-redis/dlong-*`).

| | 30s 계단 (4000) | **180s 연속 (4000)** |
|---|---|---|
| 달성 | 3,016 rps | **1,231 rps** |
| 실패 | 2.88% | **67.3% (149,263건)** |
| p50 / p95 | 681 / 2,736 ms | **4,798 / 5,001 ms** |

**판정: 아티팩트 가설 기각.** 실패 14.9만 건은 마지막 5초에 담길 수 없는 양 — 런
내내 실패했고, 노출이 길수록 악화됐다(진행성). 동반 증거: 서버측 관측 샘플러가 첫
샘플(리셋 직후) 이후 **200초 내내 /admin/stats 응답을 2초 안에 못 받았다** — admin
경로도 같은 tomcat 풀을 쓰므로, **두 노드의 워커 스레드 100개가 전부 묶인 상태가
지속**됐다는 직접 증거다. 메커니즘 가설(단정 아님): 노드당 스레드 100개가 공유
Lettuce 커넥션 1개의 파이프라인에 동기 대기 → 큐 적체가 응답을 ~5s 로 밀고, 그
동안 arrival 은 계속 쌓임. 응답을 ~5s 에서 자르는 계층은 컨테이너 로그가 정리돼
특정하지 못했다(남은 정밀도 — 결론엔 영향 없음).

**이 결과가 종합의 서사를 강화한다**: 인메모리 arm 들은 과부하에서 달성이 한계치에
고정되는데(우아한 포화), D 는 무릎을 넘기면 **시간이 지날수록 무너진다.** 나이브
외부화의 최종 청구서 항목이다.
