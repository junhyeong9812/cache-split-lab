## 1. 질문

캐시를 앱 밖(공유 Redis)으로 빼면 무엇을 얻고 무엇을 내는가.

arm D = 2 앱 노드 라운드로빈 + **공유 Redis 캐시 1개(10,000 엔트리, Lua 로 정확 LRU)**.
B 의 중복 함정도 C 의 쏠림 함정도 구조적으로 없다 — 캐시가 하나니까. 대신 **모든
요청이 매번 redis 왕복을 하고, 그 redis 는 단일 스레드**다. 실무에서 "그냥 redis 쓰면
되잖아"가 치르는 대가를 같은 리그에서 숫자로 만든다.

**Redis 기본 LRU(maxmemory)를 안 쓰는 이유** — 바이트 기준 + 샘플링 근사라 "엔트리
수 정확·이론 대조 가능" 불변식이 깨진다. Caffeine 을 버린 이유와 동일(DECISIONS §10).
HASH+ZSET+단조 카운터를 Lua 로 묶어 LruCache 와 같은 시맨틱을 구현했다.

**자원 배분 (사전 고정, 사용자 결정 "CPU 고정")**: CPU 총량 0.5 안에서 앱 0.2×2 +
redis 0.1. 근거: redis 의 일은 요청당 Lua 1회(캐시 연산만), HTTP·JSON 은 앱 몫.
캐시 10,000(공유 1개) · 스레드 100×2 · origin 4CPU·5ms · N=40,000 · s=1.5 — 전부
Phase 0/4 와 동일.

## 2. 방법 (사전 고정 — 선행 페이즈의 절차 결함 3건 반영)

- 계단: **30s** 명시 (phase-0 의 20s→30s 사후 변경을 반복하지 않는다).
  그리드 2000 / 2500 / 3000 / 3500 / 4000 / 4500 / 5000 / 6000 — 예측 무릎(3,500)
  양쪽을 500 간격으로 덮는다.
- 워밍업: **측정 시드(SEED=42)** · 2,000 RPS · 60s (phase-4 의 시드 불일치 잔재 방지).
  프리웜(전 키 1회) 선행.
- **무릎 판정 (복합 규칙 사전 등록 — 드롭 단독 기준 폐기)**: 첫 위반 계단의 직전 계단.
  위반 = ① p50 > 5ms (median 붕괴 — 스로틀 큐 신호) ② p95 > 최저 계단 p95 × 10
  ③ http_req_failed ≥ 0.1% ④ dropped > 1% (보조). A/B/C 와의 비교도 **같은 규칙을
  run2 계단표에 재적용**해 같은 잣대로 낸다.
- 60s 확인 지점: 4000·4500·5000 (phase-4 그리드와 직접 비교용).
- 나머지 규약 동일: 런 분리 · `-m 6g` 캡 · 계단 간 노드 스냅샷 · CPU 샘플링(앱·redis
  컨테이너 각각) · miss ≡ origin 교차 검증.
- 측정 전 게이트: **Lua LRU 정확성 검증**(용량 3 시나리오 — put 4개 시 최소 최근 사용
  축출, get 이 순서 승격, size ≤ capacity) 통과 후에만 측정 진입.

## 3. 사전 등록된 예측

**P1 (연역 대조 — 리그 정합성 게이트)**: D 의 정상상태 히트율 ≡ A (±0.3%p).
같은 지점 A 실측 0.9993~0.9994. 캐시가 총량 하나라 반드시 같아야 하고, 어긋나면
결과 해석이 아니라 **구현 결함 수색**에 들어간다.

**P2 (지연 바닥 — 외부화의 상시 세금)**: 저부하(3000 계단, 30s)에서
**D p50 = 0.5~1.5ms** (A 동일 계단 0.356ms 의 1.5~4배). 근거: 도커 브리지 왕복
+ Lua 실행 + Lettuce 직렬화가 매 요청에 붙는다. 히트일수록 세금 비중이 크다 —
인메모리 히트는 ~0.3ms, redis 히트는 왕복을 못 피한다.

**P3 (무릎과 병목)**: 무릎_D = **3,000~4,000** (점 추정 3,500), 무릎_B(4500)보다 낮다.
유도: 앱 노드 0.2CPU 는 B 노드(0.25)의 80% → 앱 한계 ≈ 3,600, 여기에 요청당 redis
왕복의 앱 측 CPU 추가분이 얹힌다. **병목은 앱 CPU 지 redis 가 아니다** — redis 는
요청당 Lua 1회(~수십 µs)라 0.1 코어로도 수천 ops/s 를 소화한다. 판정: 무릎 시점
앱 컨테이너 ≥ 쿼터의 90%, redis < 쿼터의 70%. (redis 가 먼저 포화하면 그것대로
발견 — "외부화는 캐시 계층 자체가 병목이 된다"로 서사가 더 세진다.)

**P4 (분포·무함정 확인)**: 전 계단 노드 분포 0.50/0.50 (±0.01) — RR 이므로.
그리고 D 는 C 형 media 붕괴(특정 노드 쏠림)가 **없어야** 한다.

**판정 기준**: P1 ±0.3%p / P2 구간 [0.5, 1.5]ms / P3 무릎 구간 [3000, 4000] +
병목 CPU 조건 / P4 ±0.01. 각 부등식 그대로 — 지표 사후 교체 금지(phase-4 교훈).
