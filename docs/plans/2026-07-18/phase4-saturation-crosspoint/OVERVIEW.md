# OVERVIEW: Phase 0·4 — 포화점과 부하 구간 우열 지도 (s=1.5)

## 주요 포인트 (5)

- 부하 생성기를 두 번 죽인 결함의 진짜 원인을 특정하고 고쳤다 — 핵심은 SharedArray 의
  **원소 단위 복사** 의미론. → 메커니즘 `TECHNICAL §개념 1`, 선택 `changelog J-1`
- 포화 측정 규약을 "멀티 시나리오 1런"에서 **k6 런 분리 계단 + 컨테이너 메모리 캡**으로
  교체했다 — 최악 실패가 "호스트 소실"에서 "컨테이너 사망"으로 격하. → `changelog J-2`, spec §3.1
- 세 arm 의 무릎을 쟀다: **A=5000 / B=4500 / C=4000** (CPU 0.5 총량). C 무릎에서 핫 노드
  부하 = 노드 용량(오차 0.5%) — 쏠림이 곧 용량 손실. → `TECHNICAL §동작 방식`, phase-0 gate
- 80~120% 부하 지도에서 **역전 실측**: s=1.5 에선 히트율 축이 평평해지고(A−B ≤ 0.4%p)
  용량·지연 축에서 C(키 해시)가 먼저 무너진다 — p50 42~104배. → phase-4 gate
- 사전 등록 예측 8개 중 4개가 빗나갔고 전부 원인이 특정됐다(분할 가정·드롭 기준·워밍업
  시드) — 빗나간 예측이 측정 절차 결함 3건을 드러냈다. → phase-0·4 gate, TIMELINE
