# TECHNICAL: Phase 0·4 — 포화 측정의 동작 모델

## 알아야 하는 개념 (구현 전제 지식)

### 개념 1: k6 SharedArray 의 원소 단위 복사

① SharedArray 는 init 컨텍스트에서 한 번 만든 배열을 전 VU 가 공유하되, **원소를 읽는
순간 그 원소만 역직렬화해 VU 쪽 사본을 만드는** 읽기 전용 자료구조다. ② 수천 VU 가 같은
40k 원소 CDF 로 이분탐색해야 하므로 "구축 1회 + 접근당 스칼라 복사"가 유일하게 메모리
상수인 배치다. ③ 모르면 "전체 배열을 원소 1개로 감싸는" 우회가 그럴듯해 보인다 —
접근 한 번에 배열 전체가 VU 로 복사돼 공유가 무효화되고, VU 수 × 배열 크기로 메모리가
곱해진다(이 프로젝트에서 부하 생성기 호스트를 동결시킨 결함).

### 개념 2: constant-arrival-rate 와 VU 수요 = rate × 지연 (Little's law)

① 도착률을 고정하는 open-model 실행기 — 서버가 느려져도 요청을 계속 보낸다(closed
model 의 coordinated omission 회피). ② 동시에 떠 있는 요청 수 = 도착률 × 응답시간이므로,
필요 VU 는 지연의 함수다: 25k RPS 라도 지연 1ms 면 VU ~25, 지연이 200ms 로 튀면 5,000.
③ 모르면 preAllocatedVUs 를 "충분히 크게" 잡는 방어가 시나리오 수만큼 곱해져 초기화
단계에서 램을 다 먹는다 — 방어가 흉기가 된다.

### 개념 3: CFS 쿼터 스로틀링이 만드는 무릎

① `--cpus 0.25` 는 100ms 주기마다 25ms 의 CPU 시간 예산 — 소진하면 다음 주기까지
전 스레드가 정지한다. ② 이 실험의 "무릎"은 이 메커니즘이 만든다: 예산 내에선 p50
0.3ms, 예산을 넘는 도착률부턴 요청이 스로틀 정지 구간에 걸려 p50 이 수십 ms 로 점프.
③ 모르면 "CPU 70% 인데 왜 느리지"를 이해할 수 없다 — 평균 사용률은 주기 내 정지를
가린다(무릎 판정은 평균 CPU 가 아니라 지연 분포로 해야 하는 이유).

### 개념 4: 드롭(dropped_iterations)은 창 길이·VU 여유의 함수, 지연은 상태의 함수

① k6 는 보낼 VU 가 없을 때만 반복을 드롭한다 — VU 여유가 있으면 지연을 늘리며 도착률을
지킨다. ② 같은 6000 RPS 포화에서 30s 창은 드롭 1.84%, 60s 창은 드롭 0%·p50 95ms 가
나왔다: 드롭은 큐가 상한에 닿는 타이밍에 민감한 이산 신호고, 지연 분포는 포화 상태를
연속적으로 노출한다. ③ 모르면 드롭을 판정 기준으로 등록하게 된다(이 프로젝트가 그랬다
— phase-4 gate P1 빗나감의 절반).

## 동작 방식

측정 파이프라인은 "생성기(APC) → nginx → 앱(캐시) → origin" 사슬에서 **앱만 CPU 를
조여** 무릎이 앱에서 먼저 서게 만든다(캘리브레이션: 생성기 천장 10.7k 의 70% 아래로).
계단마다 k6 를 새 프로세스로 띄우는 이유는 VU 풀이 런 종료로 해제되는 것이 시나리오
간 격리보다 확실하기 때문이고, 컨테이너 `-m 6g` 캡은 남는 결함이 있어도 죽는 단위를
컨테이너로 한정한다.

키 해시 arm 의 무릎이 앞당겨지는 산수: 노드 하나의 용량 R 은 CPU 쿼터가 정하고, 총
처리량 한계 = R / (최대 노드 부하 몫). 균등 분할(0.5)이면 2R, 쏠린 분할(0.628)이면
1.59R — 실측 오차 0.5%. 부하 몫은 접근 분포 × 해시 분할의 상수라 RPS 와 무관하다
(전 지점 0.629/0.371).

## 불변조건 / 계약

- 세 arm 의 총 자원 동일 (캐시 10,000·CPU 0.5·스레드 200) — 깨지면 비교 자체가 무의미.
- k6 CDF: VU 간 공유·접근당 스칼라 복사 — 깨지면(원소 래핑) 생성기 OOM.
- 부하 시퀀스 재현성: SEED 고정 → arm 간 동일 스트림 — 깨지면 차이가 배치 탓인지 부하
  탓인지 구분 불가.
- miss 합 ≡ origin 요청 증가분 — 어긋나면 카운터나 라우팅이 고장.

## 상태와 소유권

- 캐시 상태는 앱 프로세스 메모리에만 산다 — 컨테이너 재생성 = 캐시 소실이므로 arm 전환
  마다 프리웜+워밍업 필수. 무릎 계단은 같은 arm 안에서 캐시를 유지한 채 진행(정상상태 연속).
- 측정 카운터(hits/misses)는 앱 소유, k6 의 cache_hit 는 독립 집계 — 교차 검증용 이중 장부.

## 외부 경계와 의존성

- 생성기↔SUT 는 유선 LAN 이 스펙의 일부다 — WiFi 경로(RTT 3ms±1.6)는 sub-ms 측정에서
  장비 실격(실측: 목표 8k 에 4.6k, 드롭 42%).
- APC 에는 남의 상주 서비스(~6G)가 있다 — 생성기 가용 램은 15G 가 아니라 ~9G.
- BPC 의 ollama 는 유휴 시 무해하나 깨어나면 16코어를 먹는다 — 런마다 감시.

## 실패 모드 메커니즘

- **생성기 OOM(컨테이너)**: VU 메모리 곱셈이 캡 도달 → OOM kill(exit 137) → 그 계단만
  소실, 호스트 생존. 캡이 없으면 커널이 임의 프로세스를 죽이거나 스래싱으로 호스트가
  동결된다(2026-07-18 실사고 — SSH 까지 죽어 전원 재시작 필요).
- **측정 오염(워밍업 시드 불일치)**: 워밍업(SEED=1)이 못 데운 꼬리 키를 측정(SEED=42)
  첫 지점이 흡수 → 첫 지점 히트율만 0.4~1%p 눌림. 증상이 "낮은 RPS 지점의 히트율 하락"
  으로 나타나 "히트율이 RPS 의 함수"라는 거짓 결론을 유도할 수 있다.
- **비단조 계단 노이즈**: 20s 창에서 CFS 스로틀 진동·미스 버스트가 한 계단만 튀게 만든다
  (B 5500 드롭 2.8% → 6000 정상). 창을 30s+ 로 늘리면 평활 — 판정 기준은 바꾸지 않고
  창만 늘려 재측정하는 것이 사전 등록 정신과의 타협점.

## 함정 (이번에 확인된 비직관 동작)

- SharedArray 의 "공유"는 저장의 공유지 접근의 공유가 아니다 — 접근은 매번 복사다.
  원소 크기가 곧 접근 비용·VU 메모리다.
- 같은 포화를 30s 창은 "드롭"으로, 60s 창은 "지연"으로 보고한다 — 창 길이가 증상의
  표현형을 바꾼다.
- docker stats 의 CPU% 는 코어 1개=100% — 쿼터 0.25 컨테이너의 "25%"가 포화다.
