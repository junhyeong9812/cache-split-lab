# TECHNICAL: arm D — 외부 캐시의 동작 모델

## 알아야 하는 개념 (구현 전제 지식)

### 개념 1: Redis 의 축출은 근사다 — 정확 LRU 는 직접 만들어야 한다

① Redis maxmemory-policy(allkeys-lru 등)는 **바이트 예산** 도달 시 **무작위 표본
(기본 5개) 중 최소 사용**을 버리는 근사 LRU 다. ② 이 실험의 불변식은 "캐시 총량 =
엔트리 수 정확"이고 기준소스는 LRU 이론(Che)이라, 근사가 끼면 이론 대조가 무의미해진다.
③ 모르면 설정 한 줄(maxmemory)로 끝냈다가 **"LRU 같은 무언가"를 재고도 LRU 를 쟀다고
믿게 된다** — 그럴듯하고 틀린 숫자의 전형.

### 개념 2: Lua 스크립트 = Redis 의 임계 구역

① Redis 는 명령을 단일 스레드로 처리하고 Lua 스크립트 하나는 통째로 원자적이다.
② HGET+ZADD(승격), HSET+축출 루프를 스크립트로 묶으면 인메모리 구현의
ReentrantLock 과 등가의 임계 구역이 된다 — 다중 앱 노드가 동시에 때려도 순서가
직렬화된다. ③ 모르면 GET/ZADD 를 개별 명령으로 보내게 되고, 명령 사이에 다른 노드가
끼어들어 접근 순서가 오염된다 — 히트율이 조금씩, 조용히 틀려진다.

### 개념 3: 접근 순서는 시계가 아니라 단조 카운터로

① LRU 의 본질은 접근의 **전순서**다. ② ZSET 점수를 시계(ms)로 주면 같은 ms 의 접근이
동점 → ZPOPMIN 이 사전순으로 타이브레이크 → 특정 키가 체계적으로 먼저 축출된다.
INCR 카운터는 동점이 없다. ③ 모르면 고 RPS 에서만 나타나는 미세한 축출 편향이 생긴다
— 낮은 부하 검증으론 안 잡히는 종류의 결함.

### 개념 4: 동기 원격 호출의 비용 구조 — 지연 세금 vs CPU 세금 vs 스레드 세금

① 앱이 요청마다 redis 를 동기 호출하면 세 가지를 낸다: 왕복 지연(요청당), 직렬화
CPU(앱 몫), 대기 중 스레드 점유. ② 같은 호스트 도커 브리지에선 지연 세금은 median 에
거의 안 보였고(종단간 p50 차 +0.08ms), 대신 **예산 세금**(캐시 계층이 CPU 총량에서
자기 몫 20% 를 떼 감)과 **효율 세금**(동기 호출의 앱 측 비용 — 무릎 기준 per-CPU
−17%)이 곱해져 용량을 깎았다. ③ 모르면 "redis 왕복 = 느려짐"이라는 직관으로 지연만
보다가, 정작 용량(무릎)이 40% 깎인 것을 놓친다.

## 동작 방식

요청 경로: k6 → nginx(RR) → 앱 노드 → (매 요청) redis GET Lua → 히트면 응답 /
미스면 origin(5ms) → redis PUT Lua → 응답. 캐시 상태는 redis 프로세스 하나에만 있고
(HASH `cache:h` + ZSET `cache:z` + 카운터 `cache:seq`), 앱 노드는 상태가 없다 — 그래서
B 의 중복도 C 의 쏠림도 구조적으로 불가능하다. 각 앱은 Lettuce 커넥션 **하나**를 전
스레드(100)가 공유한다: 명령은 파이프라인으로 직렬화돼 나가고, 이 직렬화 지점이
"공유 캐시"라는 아키텍처의 본질을 그대로 드러낸다(풀을 쓰면 그 본질이 흐려진다 —
의도적 나이브 구현).

무릎의 산수: 총 CPU 0.5 중 앱이 0.4(0.2×2). 무릎 기준 앱 per-CPU 는 7,500 rps(B 는
9,000 — 동기 호출 세금 −17%) → 총 한계 0.4 × 7,500 = 3,000 = 실측 무릎. redis(0.1)는
무릎 계단에서 쿼터의 68%: 병목 직전이지만 병목은 아니다. REDIS_FRAC 을 줄이면 앱이
늘지만 redis 가 먼저 포화하는 지점이 온다 — 배분 곡선은 미해결(gate).

## 불변조건 / 계약

- 캐시 엔트리 수 ≤ capacity — PUT Lua 의 while 이 강제. 깨지면(HLEN>cap 지속) Lua
  원자성 가정이 깨진 것.
- ZSET 멤버 집합 ≡ HASH 필드 집합 — 두 자료구조는 항상 같은 키를 본다. 어긋나면
  축출이 값을 안 지우거나(누수) 없는 키를 지운다.
- hits+misses(전 노드 합) ≡ k6 요청 수, misses 합 ≡ origin 증가분 — 이중 장부.
- 두 KeyCache 구현의 히트율 동치(같은 총 용량일 때) — P1 연역이 회귀 게이트.

## 상태와 소유권

- 캐시 데이터·접근 순서·시퀀스: redis 소유(단일 출처). 앱 재시작에도 생존, redis
  재시작에 소실(persistence off — 실험 재현성엔 prewarm 이 담당).
- hits/misses 카운터: **앱 노드별** AtomicLong (러너의 노드별 수집 계약 유지).
  공유 상태(size·keys)는 조회 시 redis 에서 읽는다 — 파생값 저장 없음.
- Lua 스크립트 SHA: 앱 기동 시 scriptLoad — redis 재시작 시 NOSCRIPT 실패(알려진 함정 절).

## 외부 경계와 의존성

- 앱→redis: 도커 브리지 TCP, 신뢰(같은 compose). 실패 모드: 기동 시 연결 불가 = 빈
  기동 실패(의도) / 런타임 단절 = 요청 500 전파(나이브 앱 의도 — 재시도·폴백 없음).
- redis 6379 호스트 노출은 검증·모니터링용 — 실험 트래픽은 내부 네트워크만.

## 실패 모드 메커니즘

- **과부하 goodput 역행 (실측, 원인 미분리)**: 무릎 너머에서 달성이 목표 증가에 따라
  3.5k→1.8k 로 감소. 실패 다수가 ~5s 절단(p95 ≈ 5000ms). 후보: ① k6 gracefulStop(5s)이
  스텝말 미완 요청을 절단 — 계단당 in-flight 수천 개면 실패율을 크게 부풀림 ② 서버 측
  큐 적체(tomcat 100 스레드 전부 블로킹 + nginx 대기). 분리하려면 단일 장기 런(스텝말
  효과 제거)이 필요 — 미해결로 등록.
- **NOSCRIPT**: redis 가 (재)시작되면 로드된 스크립트가 사라져 evalsha 가 예외 →
  전 요청 500. 증상이 시끄러워서(침묵 아님) 실험에선 수용 — 프로덕션이면 EVAL 폴백 필요.
- **ZSET/HASH 불일치**: 이론상 Lua 원자성으로 불가. PUT 의 `old[1]==nil break` 는
  만약의 무한루프만 방지(방어선이지 복구가 아님).

## 함정 (이번에 확인된 비직관 동작)

- 외부화의 지배 비용은 왕복 지연이 아니라 **CPU 예산 분할**이었다 — 동일 호스트에선
  왕복 ~0.08ms, 그러나 무릎 −40%.
- 계단 첫 스텝의 과도 상태(VU·커넥션 초기화 + 워밍업 직후 잔재)가 저부하 스텝을
  오염시킨다 — 무릎 판정용 저부하 구간은 정상상태 재계단이 필요했다(두 페이즈 연속
  같은 교훈 — 프로토콜에 "선행 더미 스텝"을 넣는 게 맞다).
- docker stats 의 컨테이너 CPU 는 쿼터 대비가 아니라 코어 대비 % — redis "7%" 는
  한가해 보이지만 쿼터(0.1)의 70% 다.
