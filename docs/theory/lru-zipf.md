# LRU + Zipf 히트율 이론 — 이 실험의 기준소스

> 구현: `scripts/theory.py` · 검증 기록: 이 문서 §5
> 이 문서가 있어야 실험이 "돌려보고 뭐 나오나 보자"가 아니라 "이 예측이 맞나 틀리나"가 된다.

---

## 1. 왜 이론이 필요한가

이 프로젝트에는 오랫동안 구멍이 하나 있었다. 실물 리그에서 히트율 71% 가 나왔을 때
그게 맞는 숫자인지 틀린 숫자인지 판정할 방법이 없었다는 것이다. 카운터 경합 하나,
LRU 축출 버그 하나, Zipf 아닌 Zipf 생성기 하나면 그럴듯하고 확신에 찬 거짓 결론이
나오는데 — 히트율은 맞아도 71%, 틀려도 71% 처럼 보인다. 겉으로는 구분이 안 된다.

시뮬레이터를 대조군으로 두는 방법을 검토했다가 보류했는데(`DECISIONS.md` §13 참조),
그러면서 이 공백이 남았다. **LRU 를 직접 짜기로 한 결정이 이 구멍을 메웠다** —
LRU 는 이론 히트율을 계산할 수 있는 정책이기 때문이다. Caffeine 의 W-TinyLFU 를
골랐다면 이 길이 막혔을 것이다. 그때는 "분석이 된다"가 세 번째 이유였는데,
결과적으로 그게 가장 값진 이유가 됐다.

여기서 확정한 모델이 앞으로 매 페이즈의 **사전 등록된 예측값**을 만든다.
실측이 이 값과 어긋나면 셋 중 하나다 — 리그가 고장났거나, 근사의 가정이 깨졌거나,
이론이 부족하거나. 어느 쪽인지는 그때 가리면 되고, 중요한 건 **모르고 넘어가지 않는 것**이다.

## 2. 처음에 쓰려던 식은 틀렸다

설계 대화 내내 나는 예측값으로 `히트율 ≈ ln(C)/ln(N)` 을 썼다. Zipf(s=1) 에서
인기도가 `q_i ∝ 1/i` 이므로 상위 C 개의 인기도 합이 `H_C/H_N` 이고,
조화수 `H_n ≈ ln n + γ` 이므로 `≈ ln(C)/ln(N)` 이 된다는 계산이었다.

**이 식은 LRU 의 히트율이 아니다.** 이건 "상위 C 개를 **항상** 담고 있는
static oracle 캐시"의 히트율이다. 실제 LRU 는 미래를 모르니 뜨거운 키도
한동안 안 불리면 축출한다 — 그래서 항상 이보다 **나쁘다.**

문헌이 이 gap 을 정량화한다:

| 연구 | 체제 | 결과 |
|------|------|------|
| Berthet (2017) | Zipf α=1, 캐시 비율 δ=C/N 고정 | LRU miss rate 가 static 의 최대 **1.43배** (δ=0.453 에서) |
| Jelenković (1999) | 캐시 비율 → 0, tail index α>1 | LRU fault probability 가 optimal static 의 최대 **e^γ ≈ 1.78배** |

체제는 다르지만 결론은 같다 — **LRU 는 static envelope 보다 항상 낮고, 그 차이는
유계지만 무시할 수 없다.** 즉 내 봉투 뒷면 계산은 **낙관 편향**이었다.

이게 왜 위험했냐면, 낙관적인 예측을 사전 등록해두면 실측이 그보다 낮게 나올 때
"리그가 뭔가 잘못됐나?" 를 의심하게 된다. 실제로는 이론이 틀린 건데.
**빗나간 예측의 원인을 잘못된 곳에서 찾게 만드는 것** — 그게 근거 없는 예측의 진짜 비용이다.

## 3. Che's approximation

쓸 수 있는 도구는 **Che's approximation** 이다. 닫힌 형태는 아니고 1차원
root-finding 으로 푸는 수치 근사지만, LRU 의 실제 동역학을 반영한다.

### 3.1 수식

아이템 `i` 의 요청 확률을 `λ_i` 라 할 때, **characteristic time** `t_C` 는
다음 방정식의 유일한 해다:

```
Σ_i (1 - e^(-λ_i · t)) = C
```

좌변은 `t` 에 대해 단조증가하고(`t=0` 에서 0, `t→∞` 에서 N), `C < N` 이면 해가
유일하게 존재한다. 이분법으로 안정적으로 풀린다.

`t_C` 를 구하면 아이템별 히트 확률과 전체 히트율이 나온다:

```
h_i = 1 - e^(-λ_i · t_C)
h   = Σ_i λ_i · h_i
```

직관적으로 `t_C` 는 "캐시에 들어온 아이템이 축출되기까지 버티는 시간"이다.
어떤 아이템이 그 시간 안에 다시 불리면 히트, 아니면 미스다. 그래서
`h_i = P(다음 요청 간격 < t_C)` 이고, 지수분포 가정에서 위 식이 나온다.

### 3.2 가정 — 그리고 우리 실험이 왜 여기 딱 맞는가

이 근사는 **IRM (Independent Reference Model)** 을 가정한다: 매 요청이 이전 요청과
무관하게 인기도 분포에서 독립적으로 뽑힌다는 것. 실제 웹 트래픽은 시간적 지역성이
있어서 IRM 이 안 맞는 경우가 많은데 — **우리 k6 부하는 정확히 IRM 이다.**
매 요청이 독립적으로 Zipf 에서 추출된다(`k6/zipf.js`).

즉 우리는 이 근사가 가장 잘 맞는 조건에서 실험하고 있다. 이건 이 실험의 강점이자
동시에 **일반화의 한계**다 — 실제 서비스는 시간적 지역성이 있어서 우리 숫자보다
히트율이 높게 나올 것이다. 우리가 재는 건 "지역성이 전혀 없을 때의 캐시 동작"이고,
그건 하한에 가깝다.

두 번째 가정은 **아이템들의 캐시 점유가 서로 독립**이라는 것이다. 실제로는 캐시 용량이
고정이라 음의 상관이 있는데(하나가 들어오면 다른 하나가 나간다), 근사는 이를 무시한다.
이게 이 근사의 핵심 근사 지점이고, 오차의 출처다.

### 3.3 정확도

문헌은 엄밀한 worst-case 오차 상한을 제시하지 않는다. 대신:

- **균등 분포(s=0)에서는 정확(exact)하다.** 근사가 아니라 유한한 N, C 에서도
  성립하는 정확한 값이다. Fricker et al. 이 명시한다.
- **N, C → ∞ 극한에서 점근적으로 정확**하다 (Fagin 1977).
- Zipf(0.8), Zipf(1.2) 시뮬레이션과 "for all practical purposes" 일치한다고
  Fricker et al. 이 보고한다. 다만 이건 그래프 기반 경험적 검증이지 증명이 아니다.

정확도가 떨어진다고 알려진 조건은 catalog·캐시가 작을 때, 인기도가 극단적으로
소수에 집중될 때다. 우리 실험 범위(N=5,000~80,000, C=5,000~10,000, s=0~1.5)는
문헌이 검증한 범위 안에 있다.

## 4. arm 별 예측을 만드는 법

여기가 이론을 우리 실험에 붙이는 지점이다. 핵심 통찰 두 개가 있다.

**arm B 는 "캐시 C/2 짜리 단일 노드"와 통계적으로 동등하다.** 라운드로빈은 분포를
보존하기 때문이다 — 각 노드가 보는 요청 스트림은 전체 스트림과 통계적으로 같은
Zipf 분포다. 그래서 요청 하나의 입장에서 보면, 자기는 캐시 C/2 인 노드에 떨어졌고
그 노드의 캐시는 전체와 같은 분포로 채워져 있다. 두 노드가 같은 키를 중복 보유하는
현상은 이 등가성의 **결과**지 별도의 메커니즘이 아니다.

따라서 `B = che_hit_rate(N, s, C/2)`.

**arm C 는 "캐시 C/2, 키 공간 N/2 인 단일 노드"와 동등하다.** 키 해시가 키 공간을
반으로 갈라 각 노드에 전담시키므로, 노드 하나는 N/2 개의 키를 C/2 캐시로 감당한다.
비율 `(N/2)/(C/2) = N/C` 가 A 와 같다. 그래서 **C 의 히트율은 A 와 (거의) 같다.**

따라서 `C = che_hit_rate(N/2, s, C/2)`.

"거의"인 이유는 Zipf 가 완전한 스케일 불변이 아니기 때문이다. 키 공간을 반으로
자르면 분포의 모양이 미세하게 달라진다(정규화 상수 H(N/2,s) ≠ H(N,s)/2).
그래서 C 는 A 보다 아주 조금 낮게 나오고, 그 차이는 s 가 클수록 커진다.

## 5. 검증 — 이론이 실측을 맞혔다

2026-07-17, 비율 1 (N=10,000, 캐시 총량 10,000), s=1.0, 500 RPS, BPC 실물 리그.

| arm | 실측 히트율 | Che 근사 | 오차 | static envelope (내 원래 예측) |
|-----|------------|----------|------|------------------------------|
| A | 1.0000 | 1.0000 | 0.00%p | 1.0000 |
| **B** | **0.8996** | **0.8986** | **0.10%p** | 0.9292 (**+2.96%p 낙관**) |
| C | 1.0000 | 1.0000 | 0.00%p | 1.0000 |

**손실 A→B: 실측 10.04%p vs Che 10.14%p — 오차 0.10%p.**
내 원래 봉투 뒷면 계산은 7.08%p 였다.

이 표가 말하는 게 셋이다. 첫째, **리그가 제대로 돈다** — 이론이 0.1%p 이내로
맞는다는 건 LRU 구현·Zipf 생성기·카운터·라우팅이 전부 의도대로 동작한다는
간접 증거다. 둘째, **Che 근사가 우리 조건에서 유효하다** — 이제 매 지점의 예측값을
계산할 수 있다. 셋째, **내 원래 근사가 낙관 편향이라는 문헌의 지적이 실측으로 확인됐다** —
실측(0.8996)이 static envelope(0.9292)보다 낮고 Che(0.8986)에 붙었다.

세 번째가 특히 중요하다. 문헌 조사와 실측이 **독립적으로** 같은 결론에 도달했다.
문헌만 봤으면 "그런가 보다" 였고, 실측만 봤으면 "예측이 왜 빗나갔지?" 였을 것이다.
둘이 만나서야 "내 근사가 틀렸고, 이만큼 틀렸고, 이유는 이것"이 됐다.

균등 분포 검산도 통과했다 — Che 가 N=10,000 에서 C=10,000 일 때 1.0000,
C=5,000 일 때 정확히 0.5000 을 낸다. 문헌이 말한 "s=0 에서 exact" 가 재현된다.

## 7. 이 예측표가 실험 설계를 바꿨다

표를 만들고 나서야 보이는 게 둘 있었다. 둘 다 런을 돌리기 전에 알게 된 것이고,
그게 이론을 먼저 확정한 값어치다.

**첫째, s=2 는 죽은 구간이다.** 전 arm 이 99.99% 라 잴 게 없다. 쏠림이 너무 세서
뜨거운 몇 개가 작은 캐시에도 전부 들어가고, 그래서 캐시를 쪼개든 말든 아무 차이가
없다. 이건 "중복 = 기능" 명제의 극단적 확인이긴 한데, **15 런을 돌려서 확인할
가치는 없다.** 계산으로 나오는 걸 측정하지 않는다는 원칙(`DECISIONS.md` §6)이
여기 그대로 적용된다. 확인 사살로 한 지점만 돌리면 충분하다.

함정이 실제로 최대가 되는 곳은 **s ≈ 0.5** 다(비율 4 에서 15.42%p). 균등(s=0)보다
오히려 크다. 이건 반직관적인데 — 약한 쏠림이 있으면 A 의 큰 캐시가 그 쏠림을 활용해
히트율을 올리는데, B 의 작은 캐시는 같은 쏠림을 덜 활용하기 때문이다. 완전 균등에선
양쪽 다 쏠림을 활용할 게 없어서 차이가 순수하게 용량 비율로만 난다.

**둘째, B 와 C 의 교차점은 히트율에 나타나지 않는다.** 이게 더 중요한 발견이다.

| s | 함정 크기 (A→B 손실) | arm C 최대 노드 부하 | 1등 키 몫 |
|---|---------------------|---------------------|----------|
| 0 | 12.50%p | 50.2% | 0.00% |
| 0.5 | **15.42%p** (최대) | 50.4% | 0.25% |
| 1.0 | 8.25%p | 52.5% | 8.95% |
| 1.5 | 0.47%p | **69.6%** | 38.4% |
| 2.0 | 0.01%p | **83.5%** | 60.8% |
| 3.0 | — | **95.2%** | 83.2% |

(비율 4 기준. 부하 몫은 `/key/key-N` 문자열 해시로 노드를 할당해 계산 —
nginx 의 실제 해시함수와는 다르지만 "균등한 임의 분할"이라는 성질은 같다.)

두 열이 정반대로 움직인다. **함정 크기는 s 가 커지면 사라지고, C 의 부하 쏠림은
s 가 커지면 폭발한다.** s=2 면 1등 키 하나가 트래픽의 60% 를 먹고, 키 해시는 그 키를
한 노드에 고정하므로 그 노드가 84% 를 혼자 받는다. 라운드로빈이면 정확히 반씩
나눠 받았을 부하다.

그래서 **s ≈ 1.2~1.5 어딘가에서 뒤집힌다.** 아래쪽은 C 가 이기고(캐시 이득은 큰데
쏠림 비용이 0), 위쪽은 B 가 이긴다(캐시 이득이 없는데 쏠림 비용만 있다).

**그런데 이 교차가 히트율 지도(Phase 1~3)에는 안 보인다.** 고 skew 에서 B 와 C 는
둘 다 100% 라 히트율로는 구분이 안 되기 때문이다. C 의 대가는 **처리량과 지연**으로
나온다 — 핫 노드가 먼저 포화하니까. 즉 **교차점은 Phase 4(부하 구간)에서만 관측된다.**

이건 페이즈 설계의 수정을 요구한다. Phase 1~3 은 "함정의 크기"를 재는 것이고,
"B vs C 의 우열"은 Phase 4 의 질문이다. 원래 나는 이 둘을 같은 축에 놓고 있었는데,
이론이 그게 불가능하다는 걸 알려줬다.

## 8. 이 문서가 못 하는 것

Che 근사의 원논문(Che 2002, IEEE JSAC)과 Fagin 1977 은 **paywall 이라 전문을 읽지
못했다.** 수식은 이를 정면으로 재유도한 후속 논문 두 편(Fricker et al. 2012,
Martina et al. 2016 — 둘 다 arXiv 전문 확인)에서 교차검증했고, 두 논문이 동일한
형태를 제시한다. Jelenković 1999 의 `e^γ` 상한과 Berthet 2017 의 1.43 배 수치는
**초록·검색 스니펫 수준으로만 확인**했으므로 정확한 조건은 원문 재확인이 필요하다.

이 문서의 실용적 결론(Che 근사를 예측에 쓴다)은 §5 의 실측 검증으로 뒷받침되므로
문헌의 세부가 조금 달라도 흔들리지 않는다. 다만 "LRU 가 static 의 최대 1.43 배"
같은 **구체적 수치를 인용할 때는 미확인임을 밝혀야 한다.**
